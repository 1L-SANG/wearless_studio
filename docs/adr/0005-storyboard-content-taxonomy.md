# 콘티보드는 촬영 방식이 아니라 상품 설명 목적에 따라 나눈다

상태: Accepted
결정일: 2026-07-17

상세페이지를 조사한 결과, 사진의 촬영 장소나 생성 방식보다 **고객에게 무엇을 설명하는지**가 페이지 순서를 더 안정적으로 설명했다. 따라서 사용자가 보는 콘티보드는 다음 세 섹션을 기준 분류로 쓴다.

1. `benefit` — 핵심 장점
2. `fit` — 핏·코디
3. `product` — 제품 확인

세 섹션의 순서는 고정한다. 구매 정보는 사진을 만들기 위한 콘티 섹션이 아니므로 에디터에서 추가·편집한다. `같은 장소`도 별도 섹션이 아니라 핏·코디 안에서 여러 컷의 공간을 맞추는 표시다.

쇼핑몰 상품 페이지 맨 위의 대표사진 영역(상품 갤러리)은 입점 플랫폼이나 쇼핑몰 껍데기가 요구하는 UI이며, 우리가 생성하는 상세페이지 내용의 섹션으로 세지 않는다. 생성 콘텐츠의 `첫 장면(hero)`은 상품 갤러리와 다른 값으로, `핵심 장점` 섹션 안에 속한다. 따라서 페이지 전체를 설명할 때는 `핵심 장점 → 핏·코디 → 제품 확인 → 구매 정보`의 네 단계이고, 콘티보드에는 사진을 만드는 앞의 세 섹션만 보인다.

상의·하의·아우터·원피스에 서로 다른 페이지 구조를 만들지 않는다. 모두 같은 세 콘티 섹션과 네 단계 페이지 흐름을 쓰고, 의류 종류에 따라 필요한 사진 조건과 생성 옵션만 달라진다.

각 콘티 카드에는 더 구체적인 설명 목적 `contentRole`을 둔다.

- `hero` — 첫 장면
- `benefit` — 핵심 장점
- `coordination` — 코디 활용
- `fit` — 핏 확인
- `realWear` — 실제 착용 느낌
- `productOverview` — 제품 전체
- `detail` — 디테일
- `custom` — 직접 구성

`custom`은 출처가 아니다. 직접 업로드한 이미지는 `source='mine'`, `contentRole='custom'`, `cutType=null`로 저장하고, 어느 큰 섹션에 놓였는지는 `sectionRole`로 따로 정한다. 출처와 사진 목적은 서로 다른 축이지만, 현재 UI에서는 내 이미지에 다른 AI 사진 목적을 고르게 하지 않는다.

`CutType = styling | horizon | product | mirror`는 없애지 않는다. 이 값은 AI가 이미지를 만드는 **내부 생성 레시피**로만 사용하고, 콘티의 탭이나 섹션 이름으로 보여주지 않는다. 기본 사진 목적이 레시피를 정하고, 생성예시는 그 레시피 안에서 가능한 구도·분위기를 고른다. 이때 목표 상품의 의류 종류가 생성예시의 적용 범위에 포함되어야 한다(ADR-0006). `custom`은 기존 레시피를 유지하거나 선택한 생성예시에 맞는 유효 레시피를 쓴다.

사진 양은 `basic | extended` 두 가지다. 둘 다 같은 세 섹션을 쓰고, 확장형은 사진 수만 늘린다. 이 두 값 외에는 읽기와 쓰기 모두 허용하지 않는다.

제품 이미지의 세부 방식은 `ghost | detail`만 쓴다. 플랫레이는 고스트샷의 생성예시 주도 표현 방식으로 흡수하고 `hanger`는 없앤다(ADR-0008). `detail`은 어느 색상이든 사용자가 올린 `Detail` 사진이 하나 이상 있으면 만들 수 있다. 목표 색상에 `Detail`이 없으면 기준색, 그다음 `Detail` 보유 첫 색상의 사진을 구조·재질 근거로 쓰고, 목표 색상명과 가능하면 swatch hex를 프롬프트에 명시해 색만 목표 색상군으로 전환한다. 단추·지퍼·스티치와 원단의 구조·형태·재질은 근거 사진 그대로 유지하며, 보이지 않는 안감·부자재·봉제 디테일은 지어내지 않는다(2026-07-17 오너 결정).

## 다른 안과 비교

- 스타일링·호리존·제품·거울을 그대로 사용자 섹션으로 쓰는 안은 “무엇을 설명하는가”와 “어떻게 찍는가”가 섞여 페이지 흐름이 흔들리므로 채택하지 않았다.
- 내부 `CutType`까지 없애는 안은 생성 옵션과 역할별 레시피 구분을 불필요하게 깨뜨리므로 채택하지 않았다.
- 기본형과 확장형을 서로 다른 페이지 구조로 만드는 안은 같은 상품 설명 순서를 두 벌로 관리하게 되므로 채택하지 않았다.
- 아뜨랑스처럼 아주 긴 룩북은 **분량만** 아웃라이어로 본다. 내부의 장면 순서와 섹션 구성은 비교 근거로 계속 쓰되, 긴 룩북 전용 제3 구조나 제3 사진 양 프리셋은 만들지 않는다.

## 영향

- 새 콘티는 `taxonomyVersion: 2`, `sectionRole`, `contentRole`을 저장한다.
- ADR-0003과 ADR-0004의 생성 레시피·거울샷·레퍼런스 규칙은 계속 유효하다. 다만 `BlockKind`를 현재 사용자 섹션으로 본 설명, 거울샷을 `kind='styling'`으로 저장한다는 설명, 컷 종류를 사용자 탭이나 섹션 이름으로 보여준다는 설명은 이 결정으로 대체한다.
- 레거시 호환은 제공하지 않는다. 전환 시점 프로젝트는 전부 테스트 데이터로, 리셋한다(2026-07-17 오너 결정).
- 카피 작성과 페이지 조립은 `contentRole`을 정본으로 사용하고, 누락 시 `cutType`으로만 방어적으로 정규화한다.

## 2026-07-19 후속 결정 — contentRole은 내부 전용이다

`contentRole`은 레시피 파생, 카피 방향, 입력 게이트와 카드 정렬을 위한 내부 개념으로 유지한다. 사용자에게 `hero`, `benefit`, `fit` 같은 역할 이름을 직접 고르게 하는 역할 칩 UI는 폐지하는 방향으로 간다. 사용자의 선택은 생성예시 사진으로 일원화하고, `hero`는 핵심 장점 섹션의 첫 카드에 자동 부여한다.

이 UI 개편은 생성예시 실제 배선과 한 패키지로 진행하며, 비어 있는 생성예시 버킷을 먼저 보강한다. 현행 역할 칩 UI는 이번 결정 기록 작업에서는 변경하지 않는다.
