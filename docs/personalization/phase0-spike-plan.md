## 3. 정량 게이트 정의 + 판정

평가 순서는 항상 **정책 → 정량**이다. 정책 게이트를 통과하지 못한 경로는 아래 정량 절차를 아예 실행하지 않는다.

### 3.1 정책 게이트 (선행, pass/fail)

- 확인 항목: (1) 공급자 약관이 상업적 실인물(real-person) likeness 생성을 허용하는가, (2) person-generation 관련 지역 제약(Gemini AUP/Vertex 정책, Replicate ToS)이 서비스 대상 지역(한국)에 적용되는가.
- 방법: 각 공급자의 이용약관·AUP·모델 카드를 직접 확인(T0-1 라이선스 확인과 동일 엄격성). 결과는 "허용/조건부/불허" 3분류로 기록.
- 판정: 하나라도 "불허"면 해당 경로 즉시 컷, §2 표의 정량 컬럼은 "해당없음(정책 컷)"으로 남긴다.

### 3.2 얼굴 유사도 — 임계 캘리브레이션 절차

목표: ArcFace/InsightFace 코사인 유사도 임계값을 "동일인 인지 가능" 기준으로 확정한다(계획서 §6a: "예: ≥0.6은 캘리브레이션 예시, 절대 수치는 팀원 셀프 데이터로 확정").

1. **데이터 수집**: 팀원 본인/합성 얼굴로 (a) 동일인 쌍(같은 사람의 서로 다른 사진 2장 이상), (b) 타인 쌍(서로 다른 사람의 사진)을 준비 — T0-2 미니 게이트(동의서·파기 규칙) 완료 후에만.
2. **정량 측정**: `scripts/personalization_spike.py`의 InsightFace 임베딩으로 각 쌍의 코사인 유사도를 계산해 동일인 쌍 분포 vs 타인 쌍 분포를 확보.
3. **정성 캘리브레이션**: 동일한 쌍들을 팀원 2~3인에게 블라인드로 제시하고 "같은 사람처럼 보이는가"를 예/아니오로 판정받는다.
4. **임계 확정**: 정성 판정 "예"의 하한 코사인값 부근에서, 정량 분포(동일인 vs 타인)를 가장 잘 분리하는 지점을 임계값으로 채택. 확정 전까지 §2 표·PRD의 임계값은 전부 "TBD(예시 ≥0.6)"로 표기.
5. 이 임계값을 α/β/γ 각 경로의 "레퍼런스 vs 생성 결과" 유사도 판정에 동일하게 적용한다.

### 3.3 신원-속성 분리 (Δcos)

목표: 키/몸무게/체형 프롬프트를 바꿔도 얼굴 identity가 유지되는지 검증한다(제로샷 ID 주입의 알려진 실패 모드 — 계획서 §6a).

- 절차: 동일 얼굴 레퍼런스 세트로 신체 파라미터만 다른 2회 생성(예: 체형 "보통" vs "근육")을 수행 → 각 생성 결과와 레퍼런스 간 코사인 유사도(simA, simB)를 계산 → `Δcos = |simA − simB|`.
- 판정: Δcos ≤ 임계(임계값은 §3.2 캘리브레이션과 함께 스파이크 실측 후 확정) → 신체 속성 변경이 얼굴 identity를 훼손하지 않음. 초과 시 해당 경로는 "속성 변경 시 identity 붕괴" 실패로 기록.
- `scripts/personalization_spike.py`가 이 절차를 코드로 배선해 둔다(§6 실행 가이드 참고).

### 3.4 지연 p50/p95

- 절차: 각 경로에 대해 동일 조건(동일 레퍼런스·동일 프롬프트)으로 N≥10회 반복 호출, 각 호출의 wall-clock 지연을 기록 → p50/p95 산출.
- PRD NFR-2의 잠정치(p50 ≤30s, p95 ≤90s)와 대조해 통과/미달 판정, 실측치로 잠정치를 대체.

### 3.5 컷당 비용 USD

- 절차: 관리형 엔드포인트 공식 단가표(α: Gemini 이미지 해상도별 정가, β: Replicate 모델 페이지 실측 단가)를 기준으로 컷당 USD를 산출. 크레딧 원가 대비 마진이 양수인지 확인(PRD NFR-3).
- 측정 티어: **`gemini-3-pro-image`(high tier) 기준으로 비용을 산정한다.** light tier(`gemini-3.1-flash-image`) 대비는 선택 사항(계획서 T0-2 폴백 조항).

### 3.6 사진 1장 vs 2장 vs 3장 A/B

- 절차: 동일 인물의 레퍼런스 사진을 1장/2장/3장으로 바꿔가며 동일 프롬프트로 생성 → 각각의 유사도를 비교해 델타를 측정.
- 목적: 계획서 §8 오픈퀘스천("사진 1~2장으로 충분한가")을 해소 — 충분하다고 판정되면 PRD FR-1의 필수 슬롯 수(현재 3) 하향을 후속 검토한다.

### 3.7 Go/No-Go 판정 규칙

1. **정책 게이트 선통과(필수, 하드 게이트)** — 미통과 시 정량 결과와 무관하게 no-go.
2. 정책 게이트를 통과한 경로에 한해 **3대 정량 지표(얼굴 유사도, 컷당 비용, 지연) 전부 통과**.
3. 추가로 **Δcos 신원-속성분리 게이트 통과**(알려진 실패 모드이므로 별도 필수 체크 — 계획서 §6a).
4. 위 1~3을 모두 만족하는 경로가 하나 이상이면 그중 최저비용·최고유사도 경로를 권장 경로로 선정해 go. 하나도 없으면 no-go — 경로 변경(다른 β/γ 시도) 또는 기능 보류(계획서 Pre-mortem 시나리오 4·5).

## 4. 생체 안전

계획서 Principle 2·T0-2 미니 게이트·§6(c)·§6(d)를 그대로 적용한다.

- **합성/본인 얼굴만 사용. 실사용자 얼굴 절대 금지.** 팀원 얼굴을 부득이 사용할 경우 자유의사 서면 동의(목적·파기 시점 명시) 필수 — 고용주-종업원 관계로 동의의 자유성이 다퉈질 수 있음을 인지하고 강제성 없는 절차로 진행.
- **스파이크 종료 시 원본 얼굴·생성 결과·임베딩 전량 파기.** 로컬 디스크(레퍼런스 사진 디렉토리, `--out` 리포트에 포함된 이미지 등)를 포함해 예외 없이 삭제.
- **임베딩(및 그 해시)도 생체정보다.** 로그·리포트·JSON 산출물에 임베딩 벡터·해시를 기록하지 않는다 — 저장 허용 범위는 상태·집계 스코어(코사인 유사도 수치, Δcos 수치, 지연, 비용)뿐.
- 다운로드 폴더의 "모바일신분증 API 샘플" 등 이 기능과 무관한 외부 신원 자산은 본 스파이크에 절대 전용(轉用)하지 않는다.
- Downloads의 hackerton 관련 자산도 동일하게 본 스파이크 목적 외 사용 금지.

## 6. 실행 가이드 (요약)

상세 실행 방법은 `scripts/personalization_spike.py` 상단 docstring을 단일 진실원으로 한다. 요약:

1. `pip install insightface onnxruntime numpy pillow` (얼굴 임베딩 유사도 측정용).
2. 환경변수: 경로 α는 `GEMINI_API_KEY` 또는 `VERTEX_PROJECT`, 경로 β는 `REPLICATE_API_TOKEN`.
3. 합성/본인/동의된 팀원 얼굴 3장(정면/측면/45도)을 담은 디렉토리 준비 — `--faces-dir`.
4. `python3 scripts/personalization_spike.py --faces-dir <경로> --path alpha` 실행.
5. 스크립트가 사전조건 미충족 항목을 안내하고 안전 종료하면, 안내에 따라 준비물을 채운 뒤 재실행.
