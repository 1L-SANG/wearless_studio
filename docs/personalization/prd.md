# PRD — 사용자 얼굴·신체 개인화 AI 모델

- 작성일: 2026-07-15
- 상위 문서: [스코핑 계획](../../.omc/plans/user-face-personalization-scoping.md) (이하 "계획서")

## 1. 개요·문제정의

Wearless Studio는 현재 정적 기본 모델(DEFAULT_MODELS 3명)에 상품을 합성해 착장 컷을 생성한다. 셀러 본인(또는 동의된 본인 모델)의 얼굴·체형이 반영된 컷은 만들 수 없다.

문제:
1. 1인 셀러는 본인이 곧 모델이지만, 매번 실촬영하거나 남의 얼굴로 상세페이지를 만들어야 한다.
2. 기본 모델 컷은 셀러 간 차별화가 없고, 체형(키·몸무게) 차이로 실착 느낌 전달이 안 된다.
3. 얼굴은 개인정보보호법상 생체인식정보(민감정보)다. 동의·보관·파기 아키텍처 없이 이 기능을 만들면 법 위반에 노출되고, 한번 수집된 생체정보는 되돌릴 수 없다.

해법 방향(계획서 ADR 확정): **per-user 학습 없이 얼굴 1~3장으로 identity를 주입하는 제로샷 경로.** 후보 경로는 α(기존 Gemini 레퍼런스 배관 재사용) / β(관리형 Qwen-Image-Edit-2511, Qwen-Image-2512 계열 우선 검토) / γ(SDXL/FLUX+ID어댑터, 대조군) — **최종 경로는 스파이크 T0-2가 결정한다.**

## 2. 목표 / 비목표

### 목표
- G1. 사용자가 본인 얼굴 3장(정면/측면/45도)과 키·몸무게·체형을 등록하면, 기존 상품 합성 파이프라인에서 "내 모델"로 착장 컷을 생성할 수 있다.
- G2. 생체정보 수집 전 분리 동의(서비스 이용 ↔ 학습 활용) + 국외이전 동의를 강제하고, 철회·계정삭제 시 파생물(임베딩·개인화 산출물)·백업까지 연쇄 파기한다.
- G3. 품질검사(QC)로 불량 입력(가림/저해상도/다인)을 업로드 시점에 차단해 생성 실패·비용 낭비를 줄인다.
- G4. Phase 0에서 정책·품질·비용 go/no-go 데이터를 확보한 뒤에만 빌드한다.

### 비목표
- N1. **per-user LoRA/DreamBooth 파인튜닝은 초기 범위에서 제외.** 제로샷 미달 세그먼트에 한해 별도 ADR로만 재상정(계획서 T2-4).
- N2. **미성년자 지원 안 함.** 초기 릴리스는 미성년자 차단(법정대리인 동의 플로우 없음).
- N3. **Phase 2 전 실사용자 생체정보 수집 금지.** 스파이크는 본인(창업자 자기동의)/합성/서면 동의 얼굴만.
- N4. 타인 얼굴 등록(FaceMarket 라이선스 얼굴과의 목적·주체 분리 — 본 기능은 "계정 본인" 전용).
- N5. 자체 GPU 상시 운영(서버리스/관리형만), 동영상 생성, 실시간 카메라 트라이온.

## 3. 페르소나

- **P1. 1인 의류 셀러 "지수" (주 타깃).** 스마트스토어·자사몰 운영, 촬영 예산 없음, 본인이 모델. 신상 등록마다 착장 컷 5~10장 필요. 기대: 셀카 3장으로 본인 얼굴·체형이 반영된 상세컷. 우려: "내 얼굴 사진이 어디 저장되고 언제 지워지는지".
- **P2. 예비 창업자 "민호".** 판매 전 상세페이지 시안 검증 목적. 저빈도 사용, 크레딧 민감. 기대: 싸고 빠른 1회성 생성, 쉬운 탈퇴·삭제.
- **P3. 내부 운영·컴플라이언스 담당.** 동의 이력·파기 감사 로그 조회, 철회 처리 SLA 확인, 사고 시 영향 범위 산정이 가능해야 함.

## 4. 유저 스토리 + 수용 기준

- **US-1. 촬영 가이드와 3장 업로드.** 지수는 가이드를 보고 정면/측면/45도 3장을 업로드한다.
  - AC-1a: 업로드 화면에 각도별 예시·촬영 가이드(조명/무배경/무가림/1인)가 업로드 전에 노출된다.
  - AC-1b: 3개 슬롯(정면/측면/45도)이 각각 독립 상태를 가지며, 3장 모두 QC 통과 시에만 "등록 완료" 상태가 된다.
  - AC-1c: 슬롯별 재업로드가 가능하고 나머지 슬롯 상태는 유지된다.
- **US-2. QC 실패 사유 안내.** 지수의 측면 사진이 마스크로 가려져 있으면, 왜 실패했는지와 어떻게 다시 찍을지 안내받는다.
  - AC-2a: 실패 시 사유코드(가림/저해상도/다인/각도 불일치)별 한국어 안내문 + 해당 각도 재촬영 CTA가 표시된다.
  - AC-2b: QC 불합격 원본은 판정 직후 서버 메모리에서 즉시 파기된다(저장 0바이트, R2 put 없음 — api-spec §3.2와 동일 계약). 실패 사유는 업로드 응답으로만 전달되며, 어떤 시점에도 실패 이미지·파생물이 서버·스토리지에 존재하지 않음을 테스트로 검증한다.
- **US-3. 분리 동의.** 지수는 서비스 이용 동의만 하고 학습 활용 동의는 거부한 채 기능을 쓸 수 있다.
  - AC-3a: 필수(생체정보 수집·이용, 국외이전)와 선택(학습 데이터 활용) 항목이 개별 체크박스이며 사전 체크 금지.
  - AC-3b: 선택 동의를 거부해도 개인화 생성 기능 사용에 제한이 없다.
  - AC-3c: 각 동의는 항목·버전·타임스탬프로 기록되고 P3가 조회할 수 있다.
- **US-4. 국외이전 인지.** 지수는 얼굴 데이터가 해외(예: Google/Replicate = 미국)에서 처리됨을 동의 화면에서 명시적으로 확인한다.
  - AC-4a: 국외이전 동의 문안에 개인정보보호법 제28조의8 제2항의 필수 고지사항 전부 — 이전되는 항목 / 이전 국가·시기·방법 / 이전받는 자의 명칭·연락처 / 이용 목적·보유기간 / 동의 거부권과 거부 시 불이익 — 가 표기된다. 미동의 시 업로드 진행 불가(최종 문구는 §9 법무 검토 항목에 연결).
- **US-5. 신체 정보 입력.** 지수는 키·몸무게를 입력하고 체형을 고른다.
  - AC-5a: 키(cm)·몸무게(kg)는 필수이며 미입력 시 다음 단계 진행 불가. 범위 검증은 api-spec §3.3 값(키 100~230, 몸무게 30~200 — 잠정, 구현 시 확정하되 두 문서 동시 갱신)을 단일 기준으로 하며, 실패 시 인라인 오류.
  - AC-5b: 체형은 선택형 pill UI(마른/보통/근육/통통/직접입력)이며 "직접입력" 선택 시 자유 텍스트 필드가 활성화된다.
- **US-6. 개인화 생성.** 지수는 상품 이미지를 올리고 모델 선택에서 "내 모델"을 골라 착장 컷을 생성한다.
  - AC-6a: READY(= SET_COMPLETE ∧ 필수 동의(service_use, cross_border_transfer) 유효 ∧ 신체 필수값 입력 완료 — api-spec §2) 상태에서만 "내 모델" 옵션이 노출된다(미충족 시 등록 유도). 판정은 `GET /v1/personalization/status`의 canGenerate를 단일 진실원으로 참조한다.
  - AC-6b: 생성 결과의 얼굴 유사도가 스파이크에서 확정한 임계(예: ArcFace cos ≥ 임계) 이상이어야 사용자에게 노출된다(미달 시 자동 재시도 또는 실패 안내).
- **US-7. 결과 보정.** 결과가 마음에 안 들면 지수는 재생성하거나 신체 파라미터를 조정해 다시 생성한다.
  - AC-7a: 결과 화면에서 재생성이 가능하다(기존 regenerate 플로우와 정합).
  - AC-7b: 키·몸무게·체형 변경 후 재생성 시 얼굴 유사도가 유지된다(신원-속성 분리, Δcos ≤ 스파이크 확정 임계).
- **US-8. 학습 동의 철회.** 지수는 설정에서 학습 활용 동의만 철회할 수 있고, 서비스 이용은 계속된다.
  - AC-8a: 철회 즉시 해당 목적의 데이터 사용이 중단되고 학습용 사본이 파기 큐에 들어간다. 철회 이벤트가 감사 로그에 남는다.
- **US-8b. 삭제 접근성 (계정 유지 삭제).** 지수는 설정 > 내 모델 관리에서 계정 삭제 없이 얼굴·신체 데이터 전체 삭제를 실행할 수 있다.
  - AC-8b-1: 설정 내 삭제 진입점이 2뎁스 이내에 상시 노출된다.
  - AC-8b-2: 실행 시 파기 범위(원본·산출물·백업 소멸 예정일)를 고지하고, 확인 후 202/purgeJobId로 진행 상태를 표시한다(API: `POST /v1/personalization:withdraw` — api-spec §3.5).
  - AC-8b-3: 이 플로우는 FR-7 캐스케이드와 동일 경로다(별도 축약 경로 금지).
- **US-9. 계정 삭제 캐스케이드.** 민호가 계정을 삭제하면 얼굴 원본·파생물이 전부 사라진다.
  - AC-9a: 계정 삭제(또는 필수 동의 철회) 시 ① 즉시 파기 대상(원본 3장, 임베딩·캐시, 개인화 산출물)은 파기 잡 완료 시점에 0건임을 검증하고, ② 백업 사본은 즉시 삭제 대신 백업 보존주기(운영 확정값, 예: ≤30일) 경과로 완전 소멸하며 파기 잡이 backupPurgeDueAt을 감사 로그에 기록하고 동의 고지 문구와 일치시킨다(api-spec §3.5 캐스케이드 6단계(백업)와 동일 문안). 파기 실행·완료는 감사 로그에 기록된다.
  - AC-9b: "파기 완료" = AC-9a ①의 완료(= purged 전이)로 정의하며, 그 시점 이후 해당 사용자의 생체 데이터로 어떤 API도 200을 반환하지 않는다. 백업 소멸은 별도 만기 이벤트(backupPurgeDueAt)로 관리한다.
- **US-10. 미성년자 차단.** 미성년 사용자는 업로드 진입이 차단된다.
  - AC-10a: 성인 여부 확인 실패 시 업로드 플로우 진입 불가 + 사유 안내. 우회 경로(딥링크·API 직접 호출) 서버측 차단.
- **US-11. 사칭 방지 (Phase 2 필수).** 타인 사진 3장으로는 "내 모델"을 만들 수 없다.
  - AC-11a: 업로드 얼굴 = 계정 본인 검증(라이브니스 또는 소유 바인딩) 미통과 시 등록이 거부된다. 검증 방식은 FaceMarket OpenDID 자산 재사용 여부 조사(계획서 §8) 후 확정.

## 5. 기능 요구사항 (FR)

- **FR-1. 업로드 3장 상태머신.** 각 슬롯(FRONT/SIDE/ANGLE45)은 `EMPTY → UPLOADING → QC_RUNNING → QC_PASSED | QC_FAILED(reason)` 를 가지며, 세트 상태는 3슬롯 모두 QC_PASSED일 때만 `SET_COMPLETE`. QC_FAILED에서 재업로드 시 해당 슬롯만 `UPLOADING`으로 회귀. UPLOADING/QC_RUNNING/QC_FAILED는 클라이언트 세션 한정 상태(서버 비저장 — api-spec §3.2: 불합격분 저장 0, 실패 사유는 업로드 응답으로만 전달)이며 서버 영속 상태는 슬롯별 없음/QC_PASSED뿐이다. 업로드 진입 전 게이트: 필수 동의 완료(FR-3) + 성인 확인(FR-8).
- **FR-2. 품질검사 사유코드.** 확정 사유 4종을 코드화해 사유별 재업로드 안내를 제공한다(코드 표기는 서버 에러코드 컨벤션 snake_case — api-spec §3.2 `qc_reason` enum과 동일).
  - `occlusion`(가림: 마스크/선글라스/머리카락 등) → "얼굴 전체가 보이게 다시 촬영"
  - `low_resolution`(저해상도/흐림) → 최소 해상도 기준 + 재촬영 팁
  - `multiple_faces`(다인 감지) → "본인 1인만 나오게"
  - `angle_mismatch`(요청 슬롯과 촬영 각도 불일치) → "안내에 맞춰 정면/측면/45도로 재촬영"
  - 사유코드는 확장 가능한 enum으로 설계(추가 사유는 §9 오픈 퀘스천). QC는 사유별 통과/실패율을 메트릭으로 남긴다(원본·임베딩은 로그 금지, NFR-4).
- **FR-3. 동의 분리 플로우.** 업로드 전 단일 화면에서: ① 필수 — 생체인식정보 수집·이용 별도 명시 동의(포괄 동의 불가, 동의 거부권·거부 시 불이익 고지 포함), ② 필수 — 생체정보 국외이전 동의(이전되는 항목·이전 국가·시기·방법·이전받는 자의 명칭·연락처·이용 목적·보유기간·동의 거부권과 거부 시 불이익 고지 — AC-4a), ③ 선택 — 학습 데이터 활용 동의. 보관기간·삭제(백업 포함)·제3자 제공(원칙적 미제공) 고지 포함. 항목별 버전·타임스탬프 저장, 철회 API 제공. 상세 동의 매트릭스는 T0-3 산출물을 따른다.
- **FR-4. 키/몸무게/체형 입력.** 키(cm)·몸무게(kg) 필수 + 범위 검증. 체형 선택형 UI: 마른/보통/근육/통통/직접입력(자유 텍스트). 추가 검토 필드(성별·연령대·피부톤·헤어·의상사이즈)는 전부 선택 입력이며 MVP 포함/제외 라인은 T0-4에서 확정(§9). 체형→프롬프트 매핑 테이블은 스파이크 캘리브레이션 산출물을 사용.
- **FR-5. 개인화 생성.** READY(api-spec §2: 3슬롯 QC 통과 ∧ 필수 동의 granted ∧ 신체 필수값 입력) 사용자에게 기존 생성 플로우의 모델 선택에 "내 모델" 옵션 추가. SET_COMPLETE는 FR-1의 사진 세트 하위 상태일 뿐 단독 생성 게이트가 아니다. 생성 엔진은 스파이크 확정 경로(α/β/γ)를 사용하며 요청 페이로드는 [얼굴 3장 레퍼런스 + 키/몸무게/체형 프롬프트 + 상품 이미지]. 크레딧 차감은 기존 과금 흐름과 동일 지점에서 수행. 유사도 게이트 미달 결과는 노출 전 차단(AC-6b).
- **FR-6. 결과 보정.** 재생성(동일 입력) + 파라미터 조정 후 재생성(키/몸무게/체형 변경)을 지원. 신원-속성 분리 보장(AC-7b). 보정 범위(부분 인페인팅 등 추가 수단)는 엔진 확정 후 결정(§9).
- **FR-7. 삭제/철회 캐스케이드.** 트리거: 필수 동의 철회, 학습 동의 철회(학습 사본 한정), 계정 삭제, 보관기간 만료(자동). 파기 대상: R2 원본 3장, 얼굴 임베딩·중간 캐시, 개인화 산출물, 백업 사본, 외부 처리자 측 잔존 데이터(계약상 파기 확인). 파기 파이프라인은 실행·완료 감사 로그를 남기고 실패 시 재시도 큐로 회귀. 보관기간 수치는 §9(법무·제품 공동 결정) 확정 전 구현 착수 금지.
- **FR-8. 미성년자 차단.** 성인 확인 실패 시 클라이언트·서버 양쪽에서 업로드 플로우 차단(N2).
- **FR-9. 사칭 방지 게이트 (Phase 2 필수).** 업로드 얼굴 = 계정 본인 검증(라이브니스 또는 소유 바인딩). "사진 3장으로 누구든 생성" 차단. FaceMarket OpenDID 검증 흐름 재사용 가능성 조사 선행.
- **FR-10. 저장·접근 통제.** 얼굴 원본·산출물은 R2 비공개 저장 + 인증 게이트로만 열람(공개 URL 금지) — FaceMarket 라이선스 얼굴 선례(`src/features/model/ModelLicense.jsx`, `server/app/facemarket.py`)와 동일 패턴. API는 기존 `/v1` prefix(`server/app/routes.py`) 규약을 따른다.

## 6. 비기능 요구사항 (NFR)

- **NFR-1. 컴플라이언스.** 계획서 [§6(c) 법무 체크리스트](../../.omc/plans/user-face-personalization-scoping.md#c-법무-체크리스트-개인정보보호법--생체정보) 전 항목 충족이 릴리스 게이트. 요약: 민감정보 별도 명시 동의 / 서비스↔학습 동의 분리 / 보관기간 명시·만료 자동 파기 / 삭제=백업·파생물(임베딩·개인화 모델) 포함 / 제3자 제공 고지(원칙적 미제공) / 철회·계정삭제 연쇄 삭제 / 미성년자 차단 / 생체정보 국외이전 별도 동의·고지 / 저장 암호화(R2+인증 게이트)·접근 통제·감사 로그.
- **NFR-2. 성능(잠정 — T0-2 캘리브레이션 후 확정).** 개인화 생성 1컷 지연 p50 ≤ 30s, p95 ≤ 90s(잠정치, 스파이크 실측으로 대체). QC 판정 p95 ≤ 5s(업로드 UX 블로킹 구간). 확정 전 이 수치를 SLA로 커밋하지 않는다.
- **NFR-3. 비용.** 컷당 생성 비용(관리형 엔드포인트 단가) 상한 = 해당 생성에 차감되는 크레딧 원가 미만(마진 양수). 구체 상한 USD는 T0-2 실측 단가와 크레딧 단가 대비로 확정. 상한 초과 경로는 해상도·스텝 다운스케일 또는 보류(계획서 Pre-mortem 2).
- **NFR-4. 관측성 (계획서 §6d).** 수집 메트릭: QC 통과율·사유별 재업로드율, 유사도 스코어 분포, 생성 p50/p95, 컷당 비용, 실패·재시도율, 동의/철회 이벤트 로그, 파기 실행·완료 감사 로그. **금지: 로그·메트릭에 원본 생체 이미지·식별 가능 데이터 포함 금지 — 얼굴 임베딩(및 그 해시)도 생체정보이므로 로그·보존 금지. 상태·집계 스코어만 기록.**
- **NFR-5. 보안.** 생체 데이터 전 구간 암호화(전송 TLS, 저장 R2 암호화), 접근은 인증 게이트 경유만, 접근 감사 로그. 외부 처리자(엔진 공급자)로의 전송은 동의된 목적 범위로 한정.
- **NFR-6. 정책 적합성.** 채택 경로의 공급자 약관이 상업적 실인물 likeness 생성을 허용해야 하며(person-generation 지역 제약 포함), 정책 게이트 미통과 경로는 품질·비용과 무관하게 사용 금지(계획서 §6a).

## 8. 성공 지표

빌드(Phase 2) 지표 — 목표치 중 엔진 의존값은 스파이크 확정 후 대입:
- 온보딩 완료율: 업로드 시작 → SET_COMPLETE 도달 ≥ 70%.
- QC 1차 통과율 ≥ 60%, 사유 안내 후 재업로드 성공률 ≥ 80% (촬영 가이드 유효성 지표).
- 유사도 게이트 통과율: 노출 결과 중 얼굴 유사도 임계 이상 ≥ 90% (임계값 = T0-2 확정치).
- 생성 지연 p95·컷당 비용: NFR-2·3 확정치 이내.
- 재생성률(1결과당 재생성 횟수) — 품질 프록시, 베이스라인 수립 후 하향 추세.
- 컴플라이언스: 파기 요청 → 완료(백업 포함) SLA 준수 100%, 감사 로그 누락 0건, 임베딩 로그 유출 0건.
- 채택: 개인화 등록 사용자의 생성 재사용률(등록 후 2회 이상 생성 비율) — 베이스라인 수립.

Phase 0 성공 = 게이트 판정 산출(go/no-go 데이터 확보) 자체이며, 위 수치는 적용하지 않는다.
