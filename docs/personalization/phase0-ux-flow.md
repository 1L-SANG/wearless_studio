## 4. 사진 매수 실험 설계 (T0-2 A/B)

목적: FR-1의 필수 슬롯 수(현재 3: 정면/측면/45도)를 줄일 수 있는지 검증 — 계획서 §8·PRD §9의 오픈 퀘스천 해소.

- **비교군**: (a) 1장(정면만) vs (b) 2장(정면+측면) vs (c) 3장(정면+측면+45도). 나머지 파이프라인(엔진 경로, 프롬프트, seed)은 동일하게 고정해 사진 수만 변수로 통제.
- **대상**: T0-2 스파이크의 실험 대상과 동일(팀원 본인/합성/서면동의 얼굴만, 실사용자 금지). 경로 α/β/γ 각각에 대해 (a)(b)(c) 조합 반복.
- **측정 지표**:
  1. **얼굴 유사도**(주 지표): 레퍼런스(정면 캡처) vs 생성 결과의 ArcFace 코사인 유사도 — (a)(b)(c) 간 평균·분포 델타.
  2. **신원-속성 분리 안정성**: 키/몸무게/체형 프롬프트를 바꿔 재생성했을 때 유사도 유지 폭(Δcos) — 사진 수가 적을수록 identity가 덜 견고해 무너지기 쉬운지 확인.
  3. **각도 커버리지**: 참조 사진이 적을 때 결과 이미지의 포즈·각도 다양성 요청에서 identity가 붕괴하는지 정성 3점 척도.
  4. **비용·지연**: 참조 이미지 수 증가에 따른 1컷당 생성 지연·비용 증가폭(레퍼런스 조건부 생성은 입력 이미지가 늘수록 토큰/처리량 증가 가능).
- **판정 규칙**:
  - 1장이 §6 유사도 게이트를 통과하고 (b)(c) 대비 델타가 미미하면 → 필수 슬롯 1개로 축소 유력(FR-1 상태머신 단순화).
  - 1장은 미달, 2장부터 게이트 통과 + 3장 대비 델타 미미하면 → 2슬롯(정면+측면) 채택, 45도 슬롯은 선택 입력으로 강등.
  - 사진 수를 늘려도 게이트 통과율 개선이 없으면 → 원인이 사진 매수가 아니라 엔진/전처리 문제일 가능성(별도 이슈로 분리).
- **반영 지점**: 결과는 `angle` enum 자체를 줄이기보다, 필수 여부만 조정하는 하위호환 방식을 우선 고려한다(api-spec §6 "축소 시 angle enum만 줄이면 계약 유지"). UI는 축소 확정 시 화면②의 필수 슬롯 개수만 조정하면 되도록 슬롯 컴포넌트를 반복 가능한 단위로 설계해둔다.

## 5. 접근성 — 삭제 메뉴 배치 원칙

AC-8b-1(설정 내 삭제 진입점이 2뎁스 이내 상시 노출)을 만족하기 위한 배치 원칙:

1. **위치**: 설정 > 내 모델 관리(2뎁스 고정). "계정 삭제"와 물리적으로도, 라벨상으로도 구분한다 — 얼굴·신체 데이터 삭제는 계정을 유지한 채 실행 가능한 별개 동작(US-8b).
2. **라벨링**: 메뉴명에 대상 데이터를 명시한다. "삭제"가 아니라 "얼굴·신체 데이터 삭제" — 무엇이 지워지는지 라벨만 보고 알 수 있어야 한다.
3. **상시 노출**: 프로필 상태(`draft`/`ready`)와 무관하게 메뉴는 항상 보인다. 등록을 시작하지 않은 사용자에게는 비활성 대신 "등록된 데이터 없음"으로 상태를 알려준다. `purging` 중에는 메뉴 자체가 "삭제 진행 중" 배지로 진행 상태를 보여주고 재진입해도 동일 상태를 확인할 수 있게 한다(별도 진행 화면을 새로 찾지 않아도 됨).
4. **사전 고지**: 실행 확인 `Modal`에 파기 범위(원본 3장·임베딩·산출물·백업 소멸 예정일)를 먼저 보여준 뒤 확인을 받는다(AC-8b-2) — 확인 버튼을 누르기 전에 무엇이 사라지는지 다 보이는 순서.
5. **경로 단일화**: 학습동의만 철회하는 얕은 경로와 전체 삭제하는 깊은 경로를 UI에서는 분리해 보여주되(오조작 방지), 백엔드 캐스케이드는 FR-7 하나로 통일한다(AC-8b-3) — UI가 여러 진입점을 가져도 파기 로직이 갈라지지 않게.
6. **스크린리더 라벨**: 삭제 메뉴 항목의 접근성 라벨(`aria-label` 또는 대응 속성)에도 "얼굴·신체 데이터 삭제"를 그대로 사용해, 보조기술 사용자가 목록을 순회할 때도 대상이 무엇인지 헷갈리지 않게 한다.
