# 1주 스프린트 — 해커톤 → 실서비스 품질

- 기간: 1주 (5 작업일)
- 최우선: **생성 품질** (의류 충실도·조정·다양한 종류)
- 확정: 에디터=사진 깨짐 버그, 토스=**목 연동**(실결제 아님), 생성품질 우선
- 원칙: 각 항목 "확인 필요" = 실측(골드셋/실생성)으로 근거 남기고 → 미달만 수정

---

## WS1 — 생성 품질 (Day 1~3, 최우선) 🔴

### T1. 소재→마네킹 프롬프트 실반영 검증 + 보강
**현황**: `mannequin_generate_v1.txt`에 `fabric·construction·texture` 문구 O, 근데 전용 `${material}` 없이 analysis/PRODUCT CONTEXT로 간접 주입. `render_mannequin_prompt(template, ctx, product, analysis)` 경유.
- [ ] `analysis`의 소재 필드(재질·소재)가 `render_mannequin_prompt`까지 실제 도달하는지 추적 (server/app/agents/prompts.py)
- [ ] 도달하나 렌더 결과에 소재가 흐리면 → PRODUCT CONTEXT 블록에 **소재 명시 라인 보강**
- [ ] 검증: 소재 대비 큰 옷 4종(니트/데님/실크·새틴/가죽) 각 생성 → 질감(니트 골지·데님 워싱·실크 광택·가죽 재질) 육안 반영 확인
- **완료기준**: 4종 중 소재 특징이 생성물에 명확히 드러남. 안 되는 소재는 프롬프트 수정 후 재확인.

### T2. '조정하기' 버튼 실적용 검증
**현황**: `Mannequin.jsx` 축(fit/length/cut/silhouette + 매칭핏) 조정 → 변경 시 fitProfile 재생성(adjust 잡 폐기). `mannequin.effective_fit_profile`.
- [ ] 각 축 단독 조정 → 재생성 결과에 **그 축만** 반영되나 실측 (예: 기장만 길게 → 다른 축 유지)
- [ ] 축→프롬프트 전달(`prompt_context`/precedence rule) 확인 — 선언 축이 사진 근거를 이기는지
- [ ] 매칭 하의 핏 조정도 컷 재생성에 반영되나
- **완료기준**: 4축 각각 조정이 결과에 반영, 미반영 축은 전달 경로/프롬프트 수정.

### T3. 다양한 의류종류 마네킹컷 골드셋
**현황**: 지금까지 상의 위주 실측. clothing_type별 base gender·프롬프트·QC 편차 미검증.
- [ ] 대표 8~10종 실생성: 상의/하의(팬츠)/스커트/원피스/아우터(코트·자켓)/니트/후드/셔츠
- [ ] 실패 유형 수집: 하반신 누락·비율 깨짐·종류 오인식·기장 오류 등 → clothing_type별 표로
- [ ] 빈발 실패는 프롬프트/`select_base_gender`/QC 임계 수정
- **완료기준**: 8종+ 각 1차 생성 육안 합격률 표 + 미달 종류 수정 반영.

## WS2 — 에디터 사진 깨짐 버그 (Day 3~4) 🟡

### T4. Repro + 진단
**현황**: `Editor.jsx`가 `<img src={el.src}>` 렌더, src=`/v1/assets/{id}/file`(상대→vite 프록시→백엔드). 깨짐 후보: (a) asset /file 이 auth 게이트라 `<img>`가 Bearer 못 보냄→401, (b) 생성 직후 stale src, (c) URL 해석 오류.
- [ ] 에디터에서 깨진 사진 재현 → 네트워크 탭 확인(어떤 asset이 몇 번 코드로 실패하나)
- [ ] `/v1/assets/{id}/file` 인증 요구 여부 확인(라우트) → `<img>` 직접 로드 가능한지
- **완료기준**: 깨짐 원인 1개로 특정.

### T5. 수정 + 검증
- [ ] 원인별 수정: auth면 게이트 fetch→objectURL(얼굴 썸네일 선례) 또는 공개 URL 전환 / stale면 src 갱신 / URL이면 해석 수정
- [ ] 에디터에서 생성·수정·기존 컷 전부 정상 표시 확인
- **완료기준**: 에디터 모든 사진 정상 렌더.

---

## WS3 — 토스 페이먼츠 목 연동 (Day 4~5) 🟢

### T6. Pricing '준비 중' → 토스 목 결제 흐름
**현황**: `Pricing.jsx` 구독/추가구매 UI O, 버튼 "준비 중"(PG 미연동). `/v1/pricing-plans`·크레딧 시스템 O.
- [ ] 토스 페이먼츠 위젯(SDK) **테스트/목 모드**로 결제창 띄우기 (실결제 X)
- [ ] topup(추가구매) 흐름: 상품 선택 → 결제창 → **목 성공 콜백** → 백엔드 크레딧 적립(목 승인 or 테스트 승인)
- [ ] 결제 완료 → 크레딧 잔액 반영 + 영수증/토스트
- **완료기준**: 추가구매 버튼 클릭 → 목 결제 흐름 완주 → 크레딧 실제 증가(로컬).
- **범위 밖**: 실 PG 승인·정기구독(billing key)·webhook 프로드 배선은 다음 스프린트.

## 순서·의존성
```
Day 1  T1 소재 검증+보강
Day 2  T2 조정 검증 · T3 골드셋 시작
Day 3  T3 골드셋 마감 · T4 에디터 진단
Day 4  T5 에디터 수정 · T6 토스 시작
Day 5  T6 토스 마감 · 회귀 테스트 · 정리
```
- WS1(생성)은 서로 독립 → 병렬 가능. 실측 많아 시간 변동성 큼(버퍼 필요).
- WS2·WS3 독립.

## 리스크
- 생성 품질은 **프롬프트 튜닝 = 반복 실측**이라 시간 예측 어려움. 미달 종류 많으면 T3가 늘어남.
- 토스 목: SDK 테스트 모드는 되나, "목"의 경계(어디까지 진짜 토스 위젯 / 어디부터 가짜) 실행 중 재확인.
- 에디터 깨짐은 원인에 따라 30분(URL) ~ 반나절(auth 재배선) 편차.

## 범위 밖 (이번 스프린트 아님)
- 실 PG 결제·정기구독 프로드 · OpenDID 홀더 프로드 · 얼굴 QC 골드셋 캘리브 · 매칭 의류 에셋 복원
