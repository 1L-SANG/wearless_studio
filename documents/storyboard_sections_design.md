# 콘티보드 섹션 개편 — 설계 기록 (2026-07-10)

Claude(Fable)–Codex(GPT-5.6-sol ultra) 교차 토론으로 확정, 같은 날 1차 구현 완료.
사용자(창업자) 바인딩 결정이 최상위, 그 아래 토론 수렴 사항.

> **최신 결정 안내 (2026-07-17):** 이 문서의 1~7장은 당시 구현 과정을 남긴 기록이다. 사용자에게 보이는 섹션 이름과 카드 분류는 §8과 ADR-0005가 최신 정본이다.

## 1. 바인딩 결정 (사용자)

1. 콘티보드는 **항상 블록 카드의 세로 스택** — 상세페이지 세로 순서의 미니어처. 썸네일 가로 나열식 '이미지 뷰어' UI 금지.
2. **섹션 depth=1** — 당시에는 '같은 장소 시리즈'도 하나의 섹션으로 보았다. 최신 결정에서는 같은 장소를 핏·코디 섹션 안의 배지로 바꿨다(§8).
3. 당시 섹션 종류·순서는 수집 전이라 컷 어휘 기반 잠정안(인트로/같은 장소 시리즈/스타일링/호리존/거울샷/제품)이었다. 상세페이지 조사 뒤 §8의 세 섹션으로 확정했다.
4. 드래그 자유 이동은 유지, 추가 효과는 **섹션 상단 배경 회색 변화 수준만**.
5. 접기/펼치기 채택 — 접으면 스택(덱) 형태.
6. 샷 버튼 자동화(숨김)는 기각 — 샷 타일 상시 노출 유지.

## 2. 토론 수렴 (Codex 반박 → 수용된 것)

- **반폭 카드 기각**: 570px 보드에서 2단을 카드 축소로 미러링하지 않는다. 레이아웃 칩은 헤더에, 카드는 항상 풀폭. '2단'은 추후 `1행 왼쪽/오른쪽` 슬롯 배지 방식으로(백로그).
- **레이아웃 칩은 배선된 것만 노출**: v1 = 세로 1열 + 컬러 비교(생성기 배선 포함). 장식 칩 금지.
- **섹션 앵커 ≠ 컷 예시**: 같은 장소 시리즈의 배경 기준은 섹션 앵커(추후 `anchorExampleId`), 시리즈 안 컷의 예시는 '포즈 예시'로 강등(범위 고정). 
- **refScope v1 = 전부|포즈만**: '배경만'은 공간 앵커·배경 변형 기능과 역할이 겹쳐 보류.
- 고지 문구(확정): 포즈만 = "포즈만 참고해요. 배경에 맞지 않으면 자세와 구도가 자연스럽게 조정될 수 있어요."
- 정보 rows(방향·샷·색) 상시 노출 — hover 숨김 폐기(승인 목업과 정합).
- 회색 효과는 토큰(--bg-2)만 사용. 이동은 출발·대상 섹션 모두 '직접 구성' 표시. 제자리 드롭 무시.
- 포즈를 직접 고르면 포즈-범위 예시는 효력이 없으므로 자동 정리.
- (기각) Codex의 '접기 v1 보류' 권고 — 사용자 지시 우선으로 접기 구현함.

## 7. 레퍼런스 범위 3종 확정 (2026-07-12, 실생성 스파이크로 검증)

**결론: refScope = 전부(all) | 배경만(bg) | 포즈만(pose).** 각 범위는 **전용 자산**을 첨부한다 — 텍스트 지시만으로는 픽셀 유입(포즈·소품·의류)을 막지 못한다는 것이 두 차례 실험으로 확정됐다.

| 범위 | 첨부 자산 | 스파이크 결과 |
|---|---|---|
| 전부 | 원본 예시 | 기존 동작 |
| 배경만 | **빈 무대 플레이트**(인물 제거 + 여백 확보 재구성) | 원본 첨부 시 **예시의 의류·신발이 전이**(계약 위반) + 크롭 유출 → 플레이트로 해결 |
| 포즈만 | **옷·배경 없는 중립 전신 마네킹 투명 PNG** | 원본 첨부 시 예시의 의자·가방 등 **소품·무드 전이** → 전용 마네킹으로 해결. 원본 크롭은 복사하지 않고 현재 `shot`이 프레이밍을 정함(ADR-0009) |

- **플레이트 생성 규칙(중요)**: 단순 인물 제거로는 "인물이 화면을 채운 예시"에서 구도가 무너진다(머리 잘림). 플레이트는 **"전신이 들어갈 여백 있는 빈 무대"로 재구성**해야 하며, 컷 프롬프트에 **FRAMING OVERRIDE**(요청 샷 > 예시 크롭) 가드를 함께 넣는다.
- **비용**: 누끼·플레이트는 앵커 확정 시 예시당 사전 1회 생성(총 2콜) — 셀러 생성 시점 추가 비용 0.
- **계약**: 제품컷은 배경만/포즈만이 성립하지 않아 '전부'로 강등. 같은 장소 시리즈는 배경이 세트 기준이므로 '포즈만' 강제(기존).
- **구현**: `[[REFSCOPE:bg]]` 프롬프트 섹션, 매니페스트 역할(EMPTY SET plate 명시), 자산 레지스트리 3-variant(`{all,pose,bg}`), 인스펙터 호버 3칩, 카드 '배경'/'포즈' 미니. 서버 288 pass.
- **재사용 여지**: 빈 무대 플레이트는 보류했던 '같은 장소 세트'의 배경 기준(플레이트 방식, 결정 3-B)과 동일 기술 — 그 트랙 재개 시 재사용.
- 실험 산출물: `mockups/포즈만-실험.html`, `mockups/배경만-실험.html` (스크립트: `spike/posescope/`).

## 8. 사용자 분류 최종안 (2026-07-17)

상세페이지 구조 조사와 사용자 결정을 반영해, 콘티보드는 **사진이 고객에게 무엇을 설명하는지**로 나눈다. 촬영 장소나 생성 방식은 섹션 이름으로 쓰지 않는다.

### 8.1 세 섹션과 순서

| 순서 | 저장값 | 화면 이름 | 내부로 배정되는 사진 목적 |
|---:|---|---|---|
| 1 | `benefit` | 핵심 장점 | 첫 장면(`hero`), 핵심 장점(`benefit`) |
| 2 | `fit` | 핏·코디 | 코디 활용(`coordination`), 핏 확인(`fit`), 실제 착용 느낌(`realWear`) |
| 3 | `product` | 제품 확인 | 제품 전체(`productOverview`), 디테일(`detail`) |

`contentRole`은 콘티보드의 사용자 선택이나 표시값이 아니다. 시스템이 기본 구성·현재 섹션·카드 순서로 자동 배정하고, 첫 핵심 장점 AI 카드만 `hero`로 둔다. 새 카드는 삽입 위치와 가까운 같은 섹션 AI 카드의 생성법을 물려받는다. `custom`은 직접 업로드한 이미지를 `source='mine'`, `contentRole='custom'`으로 저장할 때 사용하며, 어느 큰 섹션에 놓였는지는 `sectionRole`로 따로 정한다.

`구매 정보`는 사진 생성 섹션이 아니다. 사이즈표, 소재 혼용률, 세탁법, 배송·교환·반품, 필수 고지처럼 사진이 필요 없는 내용은 에디터에서 넣는다. 콘티에 `0장`짜리 섹션이나 빈 이미지 칸을 만들지 않는다.

### 8.2 생성 방법은 내부에 숨긴다

`styling | horizon | product | mirror`는 AI가 이미지를 만드는 내부 레시피로 유지한다. 사용자는 이 이름이나 `contentRole`을 탭·칩·카드명으로 고르지 않는다. 핏·코디에서는 세 착용 레시피의 생성예시를 한 갤러리에서 보여주고 예시 선택으로 내부값을 자동 전환한다. 제품 확인에서는 사용자에게 익숙한 샷 이름인 `고스트샷 / 디테일샷`만 보여주며 이 선택이 내부 목적을 바꾼다.

- 실제 착용 느낌은 거울 셀피 예시와 내부 `mirror` 레시피를 쓴다.
- 제품 전체는 고스트샷을 쓴다. 플랫레이는 별도 서비스 샷이 아니라 선택한 고스트 생성예시의 표현 방식이다(ADR-0008).
- 디테일은 어느 색상이든 사용자가 올린 `Detail` 사진이 하나 이상 있으면 쓴다. 목표 색상에 없으면 기준색, 그다음 Detail 보유 첫 색상의 구조·재질을 근거로 삼고 색만 목표 색상군으로 전환한다. 멀리 찍힌 사진만 보고 원단·안감·부자재를 추측하지 않는다.
- 행거샷은 새 콘티에서 제공하지 않는다.
- 아우터의 AI 착용 이미지에는 섹션과 관계없이 `아우터 열림 정도`(전체 열림·부분 열림·전체 닫힘)를 쓴다. 제품만 나온 사진과 내 이미지에는 적용하지 않는다.

상품의 `clothingType`과 생성예시 적용 범위는 분리한다(ADR-0006). 생성예시는 목표 상품 종류가 `applicableClothingTypes`에 있을 때만 보여주며, `[top, outer]`은 화면에서 `상의·아우터 공용`으로 표시한다. `upper`라는 새 상품 종류는 만들지 않고, 아우터 열림 정도도 예시 원본이 아니라 현재 상품이 아우터인지로 판단한다.

같은 장소 정보는 핏·코디 섹션 안의 카드 묶음과 배지로만 보여준다. `spaceGroupId`·`spaceVariation`·레퍼런스 범위 계약은 그대로 유지한다.

### 8.3 사진 양과 저장 데이터

- 사진 양은 기본형(`basic`)과 확장형(`extended`) 둘뿐이다.
- 두 방식 모두 핵심 장점 → 핏·코디 → 제품 확인 순서를 쓴다. 확장형은 사진 수만 늘린다.
- 사진 양은 `basic | extended` 외 값을 읽거나 저장하지 않는다.
- 새 콘티 블록은 `taxonomyVersion: 2`, `sectionRole`, `contentRole`을 저장한다.
- 레거시 분류 호환은 제공하지 않는다. 전환 시점 프로젝트는 모두 테스트 데이터로 리셋한다(2026-07-17 오너 결정).

정본 우선순위는 ADR-0005 → `common_data_contract.md` → 이 문서 §8이다. 1~7장의 옛 컷 이름과 잠정 섹션은 구현 역사를 설명할 때만 참고한다.
