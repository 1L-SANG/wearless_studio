# Wearless — 기술 스택 결정서

> **확정 경로: "Vite 우선 + 점진 도입".** 처음부터 Next.js·전면 TS로 가지 않는다. 현재 코드(빌드 없는 CDN React 프로토타입)에서 가장 현실적인 순서로 올린 뒤, 필요해질 때 단계적으로 얹는다.
> 최종 갱신: 2026-06-09

---

## 0. 현재 코드 상태 (출발점)

- `Wearless.html` = **빌드 도구 없음.** CDN React + 브라우저 Babel(`@babel/standalone`) + `window` 전역 공유.
- npm·번들러·`import/export` 없음 → **지금 코드 그대로는 어떤 npm 패키지도 못 쓴다.**
- 따라서 첫걸음은 "Next냐 Vite냐"가 아니라 **"번들러를 깐다"** 이고, 그 최소 단계가 **Vite**다.

---

## 1. 확정 결정 (7개)

1. **지금은 Vite + React로 정리한다.** (Next.js로 바로 가지 않음)
2. **Next.js로 바로 가지 않는다.** 필요 신호가 올 때 전환(§4).
3. **features 기반 구조를 유지**해 나중에 Next 이전을 쉽게 한다.
4. **React Router 도입.** (Next 라우팅과 매핑이 쉬워짐)
5. **Zustand 도입.** (에디터 상태)
6. **TanStack Query는 백엔드/Supabase 연동 시작할 때 도입.** (그 전엔 미도입)
7. **TypeScript는 전체 전환 X → `types`/`store`/API 계약부터 점진 도입.**

---

## 2. 지금 만들 스택 (1차 목표)

| 영역 | 채택 | 비고 |
|---|---|---|
| 빌드 | **Vite** | 현재 코드에서 가장 작은 한 걸음. 브라우저 Babel 제거, npm 개방 |
| UI | **React** | 컴포넌트 로직은 프로토타입에서 거의 1:1 이식 |
| 라우팅 | **React Router** | `route/step`을 라우트로. Next App Router와 구조 매핑 쉬움 |
| 전역 상태 | **Zustand** | 에디터(blocks·elements·selection·undo/redo). +immer 권장 |
| 언어 | **JS로 시작, TS 모드 ON** | Vite를 TS 지원으로 깔되 코드는 `.jsx`로 시작, `.tsx` 혼용 가능 |
| 스타일 | **CSS (+CSS Modules)** | 토큰은 `design/colors_and_type.css` 그대로. 클래스 충돌 방지 위해 Modules |

> **아직 도입 안 함(나중):** TanStack Query(백엔드 연동 시), Next.js(SEO/공개페이지 필요 시), react-moveable·react-easy-crop(에디터 이식 단계에서).

---

## 3. 점진 도입 로드맵

```
현재 (CDN + 브라우저 Babel + window 전역)
  │  ① 껍데기만 교체 — 로직 그대로, 위험 낮음
  ▼
Vite + React + React Router + Zustand   ← 1차 목표(지금)
  │  ② 에디터 이식 단계
  + react-moveable(이동·리사이즈·회전·스냅) + react-easy-crop(크롭)
  │  ③ 백엔드/Supabase 연동 시작
  + TanStack Query (api 패칭·로딩/에러/재시도·생성 job 폴링)
  │  ④ 파일 단위, 무중단
  + TypeScript (types → store → api 계약 순)
  │  ⑤ 필요 신호가 올 때만
  ▼
Next.js 전환
```

### ①에서 실제로 하는 일 (Vite로 올리기)
- `npm create vite`(React + TS 템플릿) → React를 npm으로 고정(CDN 제거)
- `window.X` 전역 공유 → **`import/export`** 로 교체
- `<script type="text/babel">` 제거 (Vite가 변환)
- `mock/api.js`·`mock/db.js`·`types/index.js`를 실제 모듈로
- CSS를 모듈로 import (`styles/*.css` → 컴포넌트별 `.module.css` 점진 분리)
- **컴포넌트 JSX·hooks 내용은 그대로** — 바뀌는 건 로딩/모듈/라우팅 껍데기

---

## 4. Next.js 전환 시점 (그 전엔 안 감)

아래 **신호가 보일 때만** 전환한다. 그 전까지는 Vite SPA로 충분(로그인 뒤 도구라 SSR 불필요).
- 🔔 마케팅/랜딩 **SEO(검색 노출)** 가 중요해질 때
- 🔔 로그인 전 **공개 페이지**(상세페이지 공유 링크 등) 필요
- 🔔 백엔드 API를 **같은 앱에 colocate** 하고 싶을 때
- 🔔 **이미지 최적화**가 절실해질 때

> React Router로 페이지를 나눠두면 → Next App Router로의 이전이 라우트 매핑 위주라 감당 가능.

---

## 5. TypeScript 점진 도입 (전환 이벤트 없음)

순서: **`types.ts` → 에디터 `store.ts` → API 계약(`lib/api`)** → 이후 만지는 파일부터 자연스럽게.
- Vite를 TS 모드로 깔고 `allowJs: true` → `.jsx`와 `.tsx` 혼용.
- `types/index.js`의 JSDoc 계약을 `types.ts`로 승격(거의 복붙). 나머지 JS도 이 타입을 힌트로 받음.
- 단순 화면은 JS로 둬도 됨. **복잡/계약 중요한 곳부터** TS.

---

## 6. 에디터 (②단계에서 집중)

- **전역 store 1개**(Zustand): blocks·elements·selection·history. 패널·캔버스·미니프리뷰가 같은 store 구독.
- Undo/Redo: Zustand + immer + 커맨드 스택, 빠른 연속 변경 coalesce (PRD §10.5).
- 직접 구현한 드래그/리사이즈/회전/스냅 → **react-moveable**로 대체(1:1 대응). 크롭은 **react-easy-crop**.
- DOM 절대배치 유지 권장(텍스트 인라인 편집·접근성·`html-to-image` 내보내기 쉬움). konva/fabric은 본격 캔버스가 필요할 때만.

---

## 7. 백엔드 (③단계 — Supabase 가정)

| 영역 | 권장 |
|---|---|
| 데이터/인증/스토리지 | **Supabase**(Postgres + Auth + Storage) — 빠른 MVP에 적합 |
| 패칭 | **TanStack Query** — `contracts/api.js` 함수를 Supabase 호출로 교체, 로딩/에러/재시도/폴링 표준화 |
| 스키마 | `contracts/types.js` 기반 테이블 설계 |
| 크레딧 | **트랜잭션 차감**(성공 시 확정, 실패 시 환불), 잔액 선검증 (PRD §12) |
| 이미지 | Supabase Storage + CDN, 업로드 메타(파일명·용량·형식) 보존 (PRD §5.3) |
| 내보내기 | 서버 렌더(Playwright) 또는 클라 `html-to-image`. 긴 PNG / 블록별 ZIP (PRD §10.16) |
| AI 생성 | 비동기 job + 진행률 스트림(SSE/폴링) (PRD §7·§9·§10.8) |

## 8. 디렉토리 구조 (features 유지 → Next 이전 쉬움)

```
src/
  main.jsx                 ← 진입(React Router)
  routes.jsx               ← 라우트 정의 (Next 전환 시 app/ 으로 매핑)
  features/
    library/  input/  analysis/  mannequin/  storyboard/
    editor/
      store.(js→ts)         ← Zustand: blocks·elements·selection·history
      panels/               ← AI·의류·이미지·텍스트·오브젝트·레이어
  components/                ← 공용 Button·Chip·Modal·Toast·Icon …
  lib/
    api/                    ← contracts/api.js 이식 (③에서 Supabase+TanStack)
    types.ts                ← contracts/types.js 승격 (TS 1순위)
  styles/  tokens.css        ← design/colors_and_type.css 이식
```

## 9. 화면 이식 순서 (쉬운 → 어려운)

1. **보관함** — 정적 리스트 + 상태 배지
2. **입력** — 폼 + 업로드(드래그앤드롭)
3. **분석** — 폼(칩·실측·강조특징)
4. **마네킹** — 생성 대기 + A/B + 조정
5. **콘티** — 카드 리스트 + 인스펙터
6. **에디터** — 마지막. store 설계 → react-moveable → 패널 → 내보내기

---

## 10. 주의 (프로토타입에서 배운 것)

- **클래스명 충돌** — `.stepper`(스테퍼 vs 진행바), `.edge.el`(핸들 vs 요소) 충돌 사례. **CSS Modules**로 원천 차단.
- **정의된 토큰만** — `--accent` 추측 사용 금지(초기 버그). 토큰 목록 고정(`design/colors_and_type.css`).
- **회색 배경 남발 금지** — 컨테이너를 회색으로 감싸지 말 것. 깊이는 그림자로만 (디자인 규칙, PRD §14).
- **스타일 객체 전역 이름 충돌** — 프로토타입의 `const styles` 전역 충돌 교훈 → 모듈 스코프로 자연 해소.
