# Wearless — 기술 스택 결정서

> **확정 경로: "Vite 우선 + mock-first + 점진 도입".** 현재 제품 구현은 Vite + React SPA를 기준으로 유지한다. Next.js와 전면 TypeScript는 필요 신호가 올 때 단계적으로 얹는다. TanStack Query는 일부 화면에 도입됐으며(부분 적용) 앱 전역으로 확장 중이다.
> 최종 갱신: 2026-06-29

---

## 0. 현재 코드 상태

- 현재 앱은 `src/` 기준 **Vite + React SPA**다.
- React, React Router, Zustand, react-moveable은 npm 의존성으로 설치되어 있다.
- `src/lib/api/index.js`가 데이터 접근 경계이며, `VITE_API_MODE=mock|http` 듀얼 어댑터로 mock 또는 실제 FastAPI 백엔드로 라우팅한다.
- FastAPI 백엔드(Railway), Supabase(Postgres + Auth), Cloudflare R2는 구축 완료. Supabase 인증(`AuthProvider`)도 연결됐다.
- TanStack Query는 **부분 도입** 완료(Pricing·Library·CreditsHistory 화면에 useQuery 사용). 앱 전역 적용은 단계적으로 확장한다.
- JavaScript/JSX가 기본이며, TypeScript는 계약/스토어/API처럼 안정성이 필요한 파일부터 점진 도입한다.
- 기존 CDN React 단일 HTML 산출물은 `reference/prototype`과 `handoff`의 참조 자료로 둔다.

---

## 1. 확정 결정 (7개)

1. **현재 구현 기준은 Vite + React SPA다.** (Next.js로 바로 가지 않음)
2. **Next.js로 바로 가지 않는다.** 필요 신호가 올 때 전환(§4).
3. **features 기반 구조를 유지**해 나중에 Next 이전을 쉽게 한다.
4. **React Router를 사용한다.** (Next 라우팅과 매핑이 쉬움)
5. **Zustand를 사용한다.** 라우트 간 유지되는 생성 플로우 상태와, 향후 에디터 공유 상태에 쓴다.
6. **TanStack Query는 부분 도입됨(Pricing·Library·CreditsHistory).** 앱 전역 적용은 백엔드 연동 화면이 늘어날수록 순차 확장한다.
7. **TypeScript는 전체 전환 X → `types`/`store`/API 계약부터 점진 도입.**

## 2. 지금 만들 스택 (1차 목표)

| 영역 | 채택 | 비고 |
|---|---|---|
| 빌드 | **Vite** | 현재 앱 기준. npm/pnpm 의존성 사용 |
| UI | **React** | 기존 프로토타입 화면 로직을 `src/features`로 이식 |
| 라우팅 | **React Router** | 생성 플로우, 보관함, 에디터 라우트 분리 |
| 클라이언트 상태 | **Zustand** | 계정, 카탈로그, 생성 플로우 선택값처럼 라우트 간 유지되는 상태 |
| 서버 상태 패칭 | **TanStack Query (부분 도입)** | Pricing·Library·CreditsHistory에 useQuery 적용. 앱 전역 확장 진행 중 |
| 언어 | **JS/JSX 기본, TS 점진 도입** | `allowJs` 유지, 계약/스토어/API부터 필요할 때 TS화 |
| 스타일 | **CSS + CSS 변수** | 토큰은 `src/styles/tokens.css` 기준. CSS Modules는 필요 시 도입 |

> **앱 전역 미완:** TanStack Query(일부 화면 도입·앱 전역 확장 중), Next.js(SEO/공개페이지 필요 시), 전면 TypeScript.

## 3. 점진 도입 로드맵

```
현재 (백엔드 연동 진행 중)
Vite + React + React Router + Zustand + 듀얼 어댑터(mock|http)
  │  ① MVP 프론트 고도화
  + 에디터 조작/저장/다운로드 완성도 개선
  │  ② 백엔드/Supabase 연동 진행 중
  + TanStack Query (부분 도입 완료; api 패칭·로딩/에러/재시도·생성 job 폴링 앱 전역 확장)
  │  ③ 파일 단위, 무중단
  + TypeScript (types → store → api 계약 순)
  │  ④ 필요 신호가 올 때만
  ▼
Next.js 전환
```

### 지금 유지할 원칙

- 화면 컴포넌트는 `src/lib/api/index.js`만 호출한다.
- 실제 백엔드가 붙기 전까지 `src/mock`이 유일한 데이터 소스다.
- TanStack Query는 Pricing·Library·CreditsHistory에 부분 도입됐다. 나머지 화면은 백엔드 연동 확장에 맞춰 점진 적용한다.
- Zustand는 라우트 간 유지해야 하는 클라이언트 상태에만 쓴다. 화면 내부 draft, hover, modal open 같은 상태는 로컬 state로 충분하다.

## 4. Next.js 전환 시점 (그 전엔 안 감)

아래 **신호가 보일 때만** 전환한다. 그 전까지는 Vite SPA로 충분(로그인 뒤 도구라 SSR 불필요).
- 🔔 마케팅/랜딩 **SEO(검색 노출)** 가 중요해질 때
- 🔔 로그인 전 **공개 페이지**(상세페이지 공유 링크 등) 필요
- 🔔 백엔드 API를 **같은 앱에 colocate** 하고 싶을 때
- 🔔 **이미지 최적화**가 절실해질 때

> React Router로 페이지를 나눠두면 → Next App Router로의 이전이 라우트 매핑 위주라 감당 가능.

---

## 5. TypeScript 점진 도입 (전환 이벤트 없음)

순서: **`src/lib/types.js` 계약 정리 → 복잡한 store/API 계약부터 `.ts` 승격** → 이후 만지는 파일부터 자연스럽게.
- 현재 `tsconfig.json`은 `allowJs: true`이며, 앱 코드는 `.jsx` 중심이다.
- `src/lib/types.js`의 JSDoc 계약은 백엔드 스키마와 API 응답 shape를 맞추는 기준으로 유지한다.
- 단순 화면은 JS로 둬도 됨. **복잡/계약 중요한 곳부터** TS.

---

## 6. 에디터 상태와 조작

- 현재 에디터 조작은 **react-moveable**을 사용한다. 이동, 리사이즈, 회전, 스냅처럼 검증된 조작은 직접 재구현하지 않는다.
- 현재 Zustand는 계정, 카탈로그, 생성 플로우 상태에 사용한다.
- 에디터의 blocks, elements, selection, history가 여러 패널과 캔버스에 걸쳐 더 복잡해지면 별도 Zustand slice 또는 store로 승격한다.
- Undo/Redo는 PRD §10.5 기준으로 빠른 연속 변경을 하나의 단계로 합치는 정책이 필요하다.
- DOM 절대배치 유지 권장(텍스트 인라인 편집·접근성·내보내기 쉬움). konva/fabric은 본격 캔버스가 필요할 때만 검토한다.

## 7. 백엔드 (FastAPI/Supabase/R2 — 구축 완료)

| 영역 | 권장 |
|---|---|
| API 서버 | **FastAPI on Railway** — AI job orchestration, 서버 검증, 크레딧 트랜잭션 |
| 데이터/인증 | **Supabase**(Postgres + Auth) — 빠른 MVP에 적합 |
| 파일 저장소 | **Cloudflare R2** — 원본 업로드, 생성 이미지, 내보내기 결과 저장 |
| 패칭 | **TanStack Query** — `src/lib/api` 함수를 실제 API 호출로 교체한 뒤 로딩/에러/재시도/폴링 표준화 |
| 스키마 | `src/lib/types.js` 기반 테이블 설계 |
| 크레딧 | **reserve-then-confirm**(시작 시 예약 → 성공분만 확정 → 실패 시 예약 해제=미차감), 잔액 선검증 (PRD §12 · backend_integration_plan §6) |
| 이미지 | R2 + CDN, 업로드 메타(파일명·용량·형식) 보존 (PRD §5.3) |
| 내보내기 | 서버 렌더(Playwright) 또는 클라 `html-to-image`. 긴 PNG / 블록별 ZIP (PRD §10.16) |
| AI 생성 | 비동기 job + 진행률 스트림(SSE/폴링) (PRD §7·§9·§10.8) |

## 8. 디렉토리 구조 (features 유지 → Next 이전 쉬움)

```
src/
  main.jsx                 ← 진입(React Router)
  App.jsx                  ← 라우트 정의
  features/
    shell/
    library/
    product-input/
    analysis/
    mannequin/
    storyboard/
    generating/
    editor/
      Editor.jsx
      EditorPanels.jsx
  components/                ← 공용 Button·IconButton·Modal·Toast·Icon …
  store/
    useAppStore.js           ← Zustand: 계정·카탈로그·생성 플로우 상태
  lib/
    api/                    ← mock API 경계. 백엔드 연동 시 구현 교체
    types.js                ← JSDoc 계약. 필요 시 TS 승격
    limits.js               ← 크레딧 단가 등 공통 제한값
  mock/                     ← 현재 목업 데이터와 비동기 API
  styles/                   ← tokens.css, app.css, features.css, moveable.css
```

## 9. 현재 이후 우선순위

1. **MVP 프론트 완성도** — 입력, 분석, 마네킹, 콘티, 생성 대기, 에디터, 보관함 흐름을 mock-first로 안정화
2. **에디터 상태 정리** — blocks/elements/history가 커지면 Zustand store로 분리
3. **내보내기 구현** — 긴 PNG / 블록별 ZIP 생성 방식 결정
4. **백엔드 계약 구체화** — `src/lib/api` 함수별 요청/응답/에러 shape 확정
5. **백엔드 연동 진행 중** — FastAPI/Supabase/R2 구축 완료, TanStack Query 부분 도입(앱 전역 확장 중)
6. **필요 파일부터 TS 승격** — `types`, `api`, store 순서

## 10. 주의 (프로토타입에서 배운 것)

- **클래스명 충돌** — `.stepper`(스테퍼 vs 진행바), `.edge.el`(핸들 vs 요소) 충돌 사례. 새 복잡 기능은 스코프가 드러나게 클래스명을 붙이고, 필요 시 CSS Modules로 분리한다.
- **정의된 토큰만** — `--accent` 추측 사용 금지(초기 버그). 토큰 목록은 `src/styles/tokens.css` 기준으로 고정한다.
- **회색 배경 남발 금지** — 컨테이너를 회색으로 감싸지 말 것. 깊이는 그림자로만 (디자인 규칙, PRD §14).
- **스타일 객체 전역 이름 충돌** — 프로토타입의 `const styles` 전역 충돌 교훈 → 모듈 스코프로 자연 해소.
