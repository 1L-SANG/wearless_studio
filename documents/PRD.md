# Wearless PRD 통합본

> 문서 상태: Vite + React SPA 전환 및 현재 `src/` 구현 반영본  
> 최종 갱신일: 2026-07-17
> 기준 자료: 기존 `wearless_prd_integrated_v2.md`, `documents/03_기술스택_결정서.md`, `package.json`, `src/App.jsx`, `src/features/*`, `src/mock/*`, `src/lib/types.js`  
> 목적: 패션 셀러가 상품 이미지를 업로드한 뒤 AI 분석, 마네킹컷 선택, 콘티보드 구성, 상세페이지 에디터 편집, 다운로드까지 수행하는 전체 서비스 요구사항을 최신 제품 흐름과 프론트엔드 구현 기준으로 정리한다.

---

## 1. 서비스 개요

Wearless는 패션 쇼핑몰 셀러를 위한 AI 상세페이지 제작 스튜디오다. 사용자는 의류 상품명과 상품 이미지를 올리고, AI 분석 결과를 확인 및 보정한 뒤, 마네킹 기반 착용 기준 컷을 선택한다. 이후 상세페이지 콘티보드를 수정하고 AI가 이미지와 카피를 생성하면, 최종 상세페이지는 캔버스형 에디터에서 직접 편집하고 다운로드한다.

서비스의 핵심 가치는 다음과 같다.

- 상품 사진 몇 장으로 상세페이지에 필요한 컷 구성을 빠르게 만든다.
- 입력 부담은 최소화하되, AI 분석 후 중요한 상품 정보는 사용자가 확인한다.
- 마네킹컷을 통해 실제 이미지 생성 전 핏과 실루엣 기준을 잡는다.
- 상세페이지 생성 전 콘티보드에서 사진의 설명 목적, 방향, 샷, 색상, 생성예시, 매칭 의류를 조정한다.
- 생성 후에는 텍스트, 이미지, 프레임, 오브젝트, 레이어를 직접 편집할 수 있다.
- 마네킹 조정, 콘티 생성, 에디터 이미지 생성 등 크레딧 소모가 있는 행동은 실행 전에 예상 크레딧을 보여준다.

### 1.1 현재 구현 기준

현재 활성 제품 구현은 Vite + React SPA다. 기존 단일 HTML, CDN React, 브라우저 Babel 기반 산출물은 `reference/prototype` 및 `handoff`의 참조 자료로 유지하고, 실제 앱 구현 기준은 `src/` 아래의 Vite 프로젝트로 본다.

구현 전제:

- 프론트엔드는 UI와 상태를 담당하며, 실제 백엔드, Supabase, 스토리지, AI 엔드포인트를 컴포넌트에서 직접 호출하지 않는다.
- 모든 데이터 흐름은 `src/lib/api/index.js`를 경유하며, 현재는 `src/mock/api.js`와 `src/mock/db.js`가 목업 백엔드 역할을 한다.
- React Router로 생성 플로우와 보관함, 에디터 진입을 관리한다.
- Zustand는 계정, 카탈로그, 생성 플로우 선택값처럼 라우트 간 유지되어야 하는 상태에 사용한다.
- Next.js 전환은 현재 요구사항이 아니다. 공개 SEO, 공유 상세페이지, 앱 내부 API colocate, 이미지 최적화 요구가 명확해질 때 별도 의사결정으로 검토한다.

## 2. 주요 사용자

### 2.1 타깃 사용자

- 패션 쇼핑몰 셀러
- 의류 중심 온라인 셀러
- 상세페이지 제작에 시간과 비용이 부담되는 1인 또는 소규모 셀러
- 직접 촬영한 사진은 있으나 상세페이지 구성과 카피 작성이 어려운 셀러
- 핵심 장점, 핏·코디, 제품 확인에 필요한 사진을 빠르게 만들고 싶은 셀러

### 2.2 사용자 니즈

- 상품 이미지 몇 장만으로 상세페이지의 사진 구성을 빠르게 만들고 싶다.
- 모델 섭외, 촬영, 상세페이지 디자인 비용을 줄이고 싶다.
- 상품의 핏, 소재, 강조 포인트를 AI가 정리해주되 최종 정보는 직접 확인하고 싶다.
- 색상별 이미지와 코디를 쉽게 보여주고 싶다.
- 생성된 결과물을 수정할 수 있는 에디터가 필요하다.
- 크레딧이 언제 얼마나 쓰이는지 실행 전에 알고 싶다.

## 3. 전체 사용자 플로우

최신 구현 기준의 생성 플로우는 다음과 같다.

```text
1. 상세페이지 제작 진입
2. 제품 정보 입력
3. 같은 페이지에서 AI 상품 분석 및 세부 정보 확인
4. 마네킹컷 생성, 선택, 조정
5. 마네킹 페이지 안에서 상세페이지 사진 양 선택
6. 콘티보드 편집
7. 콘티 확정 후 이미지 및 카피 생성
8. 생성 대기 페이지
9. 상세페이지 에디터
10. 미리보기
11. 다운로드
12. 보관함에서 결과물 재진입
```

현재 Vite SPA의 URL 기준 라우트는 `create`, `library`, `editor/:id`로 나뉜다. `create` 내부 URL 단계는 `input`, `mannequin`, `storyboard`, `generating`이며, AI 분석은 별도 라우트가 아니라 `input` 화면 안에서 인라인으로 전개된다. 에디터는 전체 화면 작업 공간이므로 `create` 내부가 아니라 `/editor/:id`에서 앱 크롬 밖으로 열린다.

사용자에게 보이는 스텝퍼는 4단계 점 형태로 단순화한다.

```text
제품 정보·분석 → 마네킹컷 → 콘티보드 → 에디터
```

`input`과 `analysis`는 첫 번째 단계로 합쳐서 보여준다. `generating`은 에디터 진입 직전 상태이므로 스텝퍼에서는 에디터 단계로 매핑한다.

## 4. 네비게이션과 보관함

### 4.1 상단 네비게이션

상단 네비게이션은 에디터를 제외한 일반 화면에서 노출한다.

```text
wearless | 상세페이지 제작 | 보관함 | 크레딧 잔액 | 플랜 | 프로필
```

요건:

- 로고는 `wearless` 소문자 워드마크를 사용한다.
- `상세페이지 제작`은 새 생성 플로우로 진입한다.
- `보관함`은 기존 결과물 목록으로 진입한다.
- 크레딧 잔액과 플랜 배지를 항상 표시한다.
- 에디터는 전체 화면 작업 공간이므로 상단 네비게이션 대신 에디터 전용 툴바를 사용한다.

### 4.2 보관함

보관함은 사용자가 생성하거나 작업 중인 상세페이지를 관리하는 공간이다.

요건:

- 로딩 스켈레톤, 빈 상태, 에러 상태, 결과 그리드 상태를 제공한다.
- 결과 카드는 커버 이미지, 제목, 블록 수, 업데이트 시간, 상태 배지를 표시한다.
- 상태는 `완료`, `생성 중`, `초안`을 가진다.
- `새 상세페이지` 버튼과 새 생성 타일을 제공한다.
- 카드를 열면 에디터로 진입한다.

## 5. 제품 정보 입력 페이지

### 5.1 화면 목적

상품명과 상품 이미지를 입력받아 AI 분석의 기준 데이터를 만든다. 최신 구현에서는 입력 페이지의 범위를 줄여, 사용자가 처음부터 많은 정보를 입력하지 않도록 한다. 의류 종류, 세부 카테고리, 성별, 핏, 소재, 실측, 강조 특징, 모델, 매칭 의류는 AI 분석 이후 확인 단계에서 다룬다.

### 5.2 입력 항목

- 상품명
- 상품 이미지

상품명은 선택 입력이다. 사용자가 비워두면 AI 분석 결과의 `suggestedName`을 상품명으로 반영한다.

### 5.3 상품 이미지 업로드

이미지는 색상 그룹 단위로 관리한다.

#### 기준 색상

기준 색상은 의류 구조와 주요 디테일을 분석하는 대표 색상이다.

업로드 각도:

```text
Front / Back / Detail / Fit
```

화면 표기는 다음처럼 한국어 라벨을 쓴다.

```text
앞면 이미지 / 뒷면 이미지 / 디테일 이미지 / 착용 이미지
```

요건:

- 2x2 업로드 wells로 각도를 한눈에 보여준다.
- 앞면 이미지는 필수다.
- 기준 색상 이미지 수는 전체 각도 합산 최대 6장이다.
- 각 well은 클릭 업로드와 드래그앤드롭을 모두 지원한다.
- 업로드된 이미지에는 파일명, 용량, 형식을 표시한다.
- 이미지는 삭제할 수 있다.

#### 추가 색상

추가 색상은 색상 옵션 표현과 컬러별 이미지 생성에 사용한다.

요건:

- 색상은 전체 최대 3개까지 등록한다.
- 추가 색상은 정면 사진이 필수다.
- 추가 색상당 최대 3장까지 업로드한다.
- 색상 삭제가 가능하다.
- 색상 선택은 12개 팔레트에서 고른다.
- 색상을 고르기 전에는 `색상 미정`으로 표시한다.

기본 팔레트:

```text
화이트 / 그레이 / 블랙 / 아이보리 / 베이지 / 브라운
레드 / 옐로우 / 그린 / 블루 / 네이비 / 핑크
```

### 5.4 입력 완료 조건

`입력 완료` 버튼은 앞면 이미지가 1장 이상 있을 때만 활성화한다. 미충족 시 사용자가 다음 행동을 알 수 있도록 하단에 안내 문구를 표시한다.

`입력 완료`를 누르면 자동으로 같은 페이지 안에서 분석 단계가 열린다. 입력 영역은 요약 카드로 접히며, 요약 카드에는 상품명, 업로드 이미지 수, 색상 수, 대표 썸네일을 보여준다. 사용자는 요약 카드를 펼쳐 입력 내용을 다시 볼 수 있다.

### 5.5 분석 로딩

분석 중에는 페이지 하단에 스켈레톤 UI를 표시한다. 분석은 자동 진행되며, 완료 후 AI 분석 결과 폼을 같은 페이지에 인라인으로 보여준다.

## 6. AI 상품 분석 및 세부 정보 확인

### 6.1 화면 목적

AI가 상품 이미지를 분석해 상품의 구조적 정보와 상세페이지 생성에 필요한 설정값을 제안한다. 사용자는 분석 결과를 직접 확인하고 수정한 뒤 마네킹컷 단계로 이동한다.

### 6.2 레이아웃

분석 결과는 여러 페이지로 분리하지 않고 하나의 큰 카드 안에 섹션을 쌓는다. 섹션 순서는 다음과 같다.

```text
기본 정보 → 소재 → 실측 정보 → 강조하고 싶은 특징 → 모델 선택 → 매칭 의류
```

세탁 안내는 분석 단계에서 제거한다. 세탁 안내는 에디터 생성 결과의 자동 안내 블록에서 다룬다.

### 6.3 기본 정보

기본 정보 섹션은 다음 항목을 가진다.

- 의류 종류
- 세부 카테고리
- 대상 성별
- 핏

의류 종류:

```text
상의 / 하의 / 아우터 / 원피스
```

세부 카테고리:

```text
상의: 티셔츠 / 맨투맨 / 셔츠 / 니트
하의: 면바지 / 트레이닝 팬츠 / 청바지 / 슬랙스 / 치마
아우터: 셔츠 / 자켓 / 가디건 / 패딩 / 코트
원피스: 세부 카테고리 없음
```

의류 종류는 사진에서 어떻게 입었는지가 아니라 판매 상품의 분류로 정한다. 셔츠는 상의와 아우터의 세부 카테고리에 모두 있을 수 있으며, 셔츠를 열어 입은 사진만으로 아우터로 바꾸지 않는다.

대상 성별:

```text
여자 / 남자
```

핏:

```text
슬림핏 / 정핏 / 세미오버 / 오버핏
```

의류 종류와 핏은 필수 항목으로 선택 해제할 수 없다. 라벨 옆에 포인트 색(Sky) `*`로 필수임을 표시한다. 세부 카테고리와 대상 성별은 선택 해제가 가능하다. 선택된 칩은 흰 배경과 Sky 테두리로 강조한다.

### 6.4 소재

소재는 칩 형태로 입력한다.

요건:

- 각 칩은 소재명과 혼용률 퍼센트를 가진다.
- 칩 클릭 시 같은 자리에서 재편집한다.
- 소재 추가 및 삭제가 가능하다.
- 혼용률 합계 100%를 권장한다.
- 합계가 100%를 넘으면 경고를 표시한다.

### 6.5 실측 정보

실측 정보는 AI가 임의 추정하지 않는다. 사용자가 직접 입력하도록 빈칸으로 제공한다.

의류별 실측 스키마:

```text
상의: 총장 / 어깨너비 / 가슴단면 / 소매길이
하의: 총장 / 허리단면 / 엉덩이단면 / 허벅지단면 / 밑위 / 밑단단면
아우터: 총장 / 어깨너비 / 가슴단면 / 소매길이
원피스: 총장 / 어깨너비 / 가슴단면 / 허리단면 / 암홀 / 소매길이
```

요건:

- 단위는 cm다.
- 일부 항목만 입력할 수 있다.
- `실측 모름` 체크박스를 제공한다.
- `실측 모름`이 켜지면 실측 입력 필드를 숨긴다.

### 6.6 강조하고 싶은 특징

강조 특징은 상세페이지 카피와 이미지 생성 방향에 쓰이는 핵심 셀링 포인트다.

요건:

- 최대 5개까지 입력한다.
- 헤더에 `N/5개` 카운터를 표시한다.
- AI 추천 특징은 자동으로 칩에 채우고 `AI 제안` 배지를 붙인다.
- 사용자는 AI 추천 칩도 삭제할 수 있다.
- 직접 추가 입력은 칩 draft 상태에서 Enter 또는 blur로 확정한다.

### 6.7 모델 선택

모델 선택은 풀폭 카드로 제공한다.

요건:

- 모델 카드에는 썸네일과 모델명을 표시한다.
- 추천 모델에는 별 아이콘을 표시한다.
- 선택된 모델은 이후 마네킹컷과 에디터 AI 생성 패널의 기본 모델이 된다.

### 6.8 매칭 의류

매칭 의류는 모든 착용 이미지에서 판매 상품과 함께 입힐 의류를 선택하는 설정이다. 첫 장면·핵심 장점·코디 활용·핏 확인·실제 착용 느낌에 동일하게 적용하고, 제품 단독컷에는 적용하지 않는다.

요건:

- 풀폭 카드로 제공한다.
- 최대 2개까지 선택할 수 있다.
- 첫 번째 선택 항목은 `메인`, 두 번째 선택 항목은 `서브`로 표시한다.
- 매칭 의류를 실제로 선택한 프로젝트(상의/아우터)에서는 마네킹 페이지의 순차 확인 스텝에 `매칭 의류 핏` 스텝이 추가된다(§7.4). 조정값은 `fitProfile.matchCut`으로 저장된다.

## 7. 마네킹컷 생성 및 선택

> UI 세부 규칙 정본: `documents/mannequin_ui_direction.md` (as-built). 이 절은 그 방향서와 동일한 흐름을 PRD 수준에서 요약한다. — *구 A/B 후보 + 우측 조정 패널 + 조정 횟수 2회 안은 폐기(2026-07).*

### 7.1 목적

마네킹컷은 실제 상세페이지 이미지 생성 전, 의류의 핏과 실루엣 기준을 잡는 단계다(페이지 명칭: **"의류 재현성 높이기"**). 페이지 최초 진입 시 마네킹컷이 **자동 생성**되고, 사용자는 핏 축(카테고리×성별 축 + 매칭 의류 핏)을 **순차적으로 확인·조정**한 뒤, 변경이 있으면 재생성으로 반영한다. 확정된 핏은 `fitProfile`(garment_ref)로 저장돼 이후 컷 생성에 재사용된다.

### 7.2 생성 대기 UX

마네킹 생성 중에는 퍼센트와 진행바를 보여주지 않는다. 숫자가 한동안 멈춰 보이면 실패로 오해하기 쉬우므로, 현재 하는 일을 쉬운 상태 문장으로만 알려준다.

상태 문장:

```text
상품의 형태를 살펴보고 있어요
마네킹컷을 정교하게 다듬고 있어요
이미지 품질을 확인하고 있어요… (40초 이상 걸릴 때)
```

중앙에는 다음 화면과 같은 3:4 프레임 안에 의류 제작 인포그래픽 애니메이션을 보여준다. `prefers-reduced-motion`에서는 완성된 정지 화면을 보여준다. 자세한 시퀀스는 `mannequin_ui_direction.md` §5가 기준이다.

### 7.3 컷 구조

마네킹컷은 **단일 컷**으로 생성한다(구 A/B 2후보안 폐기).

컷은 맨바닥에서 매번 새로 생성하지 않는다. **분석 단계에서 선택한 대상 성별이 베이스 마네킹(남/여 고정 1종)을 결정**하고, 그 베이스 위에 입력한 의류(+선택된 매칭 하의)를 입혀 생성한다.

요건:

- 가운데 큰 컷 하나로 보여준다 — **매칭 하의까지 입은 모습**. "내 옷" 태그·원본 사진 행·테두리는 두지 않는다(은은한 그림자만).
- 컷 아래에 버전 썸네일 스트립을 둔다. 재생성으로 생긴 버전이 스트립에 추가되고 자동 선택되며, 클릭으로 다른 버전을 선택할 수 있다.
- 선택된 마네킹컷이 이후 착용컷 생성 기준이 된다.

### 7.4 핏 확인·조정 (순차 스텝)

우측 패널 대신, 이미지 아래 확인 카드에서 **순차 확인 스텝**으로 조정한다.

- 스텝 구성 = 카테고리×성별 핏 축(예: 핏·기장) + (상의/아우터 카테고리이면서 매칭 의류를 실제 선택한 경우) `매칭 의류 핏` 스텝.
- 각 스텝은 "OO을(를) 조정할까요?" 질문에 **[유지하기] / [조정하기]**로 답한다. `조정하기`를 누르면 이미지 오른쪽에 예시 타일이 세로로 나타나고, 타일을 선택해 값을 확정한다.
- 예시 타일은 참고용이다 — "선택됨" 상태를 표시하지 않는다.
- AI 추정값을 "선택된 것처럼" 노출하지 않는다. `fitProfile.axes` 초깃값이 있어도 UI에는 보이지 않는다.
- 카드 상단에 확인 항목 칩을 둔다 — 완료 = 파랑 채움 칩(✓ + [수정]), 예정 = 고스트 칩. [수정]으로 완료한 스텝을 다시 조정할 수 있다.
- **조정 횟수 제한은 없다.** 대신 조정 반영은 재생성(크레딧 소모, §7.5)으로만 이뤄진다.
- 매칭 의류 핏 조정값은 `fitProfile.matchCut`에 저장되며, 마찬가지로 재생성(유료)으로 반영된다.

### 7.5 재생성

모든 스텝 확인 후 변경이 1건 이상이면 최종 카드에 재생성 CTA를 제공한다.

요건:

- CTA 라벨에 소모 크레딧을 명시한다: `수정사항 반영하여 재생성 · 2 크레딧` (`CREDIT_COSTS.mannequinGenerate = 2`, 백엔드 `credit_cost_mannequin_generate` 미러).
- 재생성은 `fitProfile` 전체(축 + matchCut)를 반영해 새 버전을 생성한다. 새 버전은 스트립에 추가되고 자동 선택되며, 스텝은 리셋되어 재확인 루프로 돌아간다.
- 변경이 0건이면 재생성 없이 사진 양 선택(§7.6)으로 진행한다.
- 이중 제출을 방지한다(제출 중 버튼 비활성).

### 7.6 사진 양 선택 통합

별도 선택 페이지는 두지 않는다. 마네킹 페이지의 최종 카드에서 **[기본형 / 확장형]** 중 사진 양을 고르고, `이 사진 양으로 만들기`를 눌러 콘티보드로 이동한다. 두 방식은 페이지 구조가 같고 사진 수만 다르다.

| 사진 양 | 목적 | 예상 컷 수 | 공통 흐름 |
|---|---|---:|---|
| 기본형 | 대표 색상을 중심으로 꼭 필요한 사진 구성 | 12컷 | 핵심 장점 → 핏·코디 → 제품 확인 |
| 확장형 | 여러 색상과 활용 모습을 더 풍부하게 구성 | 약 13~29컷 | 핵심 장점 → 핏·코디 → 제품 확인 |

요건:

- 기본값은 `기본형`이다.
- 확장형은 섹션을 더 만들지 않는다. 같은 세 섹션 안에서 색상·핏·코디·제품 확인 사진을 더 넣는다.
- 첫 사진은 상품의 매력을 빠르게 보여주는 첫 장면이다. 정면·뒷면·사이드와 상품 전체·디테일의 필요 여부는 카테고리와 입력 근거에 맞춰 정한다.
- 같은 장소에서 이어지는 컷은 핏·코디 섹션 안에서 `spaceGroupId`로 묶는다. 같은 장소는 별도 섹션이 아니다 (ADR-0004, ADR-0005).
- 사용자가 콘티를 수정하기 전에는 사진 양을 바꾸면 기본 콘티를 다시 구성할 수 있다.
- 확장형은 색상이 하나여도 선택할 수 있다. 여러 코디·핏·제품 확인 사진을 늘리는 데 사용할 수 있다.
- 간단형은 제공하지 않는다. 사용자는 기본형과 확장형 중에서만 고른다. 이전 저장값의 기술적 호환 규칙은 `common_data_contract.md`가 다룬다.
- 카피라이팅 토글은 이 카드가 아니라 콘티보드 하단 액션바에서 제공한다.

### 7.7 크레딧 예고

마네킹 단계에서 크레딧 소모가 있는 버튼은 실행 전 예상 소모량을 버튼 라벨에 포함한다.

예시:

```text
수정사항 반영하여 재생성 · 2 크레딧
```

> 구 `의류 조정하기`(mannequinAdjust) 과금 항목은 폐기됐다(비용 0, deprecated). 조정은 전부 재생성으로 통합.

마네킹컷 최초 생성은 마네킹 페이지 진입 시 자동으로 시작·차감되므로, 차감 직전의 마지막 사용자 행동인 **분석 페이지의 `의류정보 확정 완료` 버튼에 예상 크레딧을 표시**한다(예: `의류정보 확정 완료 · 2 크레딧`).

실제 단가는 `creditCosts` 설정값으로 관리하며, 최종 정책 확정 전까지 숫자는 mock 구현용 임시값으로 본다.

## 8. 콘티보드

### 8.1 목적

콘티보드는 상세페이지 생성 전에 어떤 컷을 어떤 순서와 조건으로 만들지 정하는 단계다. 이 단계의 썸네일은 최종 이미지가 아니라 생성 방향을 잡기 위한 예시다.

### 8.2 진입 레이아웃

처음 진입하면 카드 리스트만 가운데 정렬한다. 사용자가 카드를 선택하면 화면이 좌/우 분할 레이아웃으로 바뀐다.

```text
좌측: 블록 카드 리스트
우측: 선택 블록 인스펙터
하단: 고정 액션바
```

선택을 해제해도 한 번 열린 분할 레이아웃은 유지하며, 우측은 빈 상태와 `내 이미지 업로드` 버튼을 보여준다.

### 8.3 블록 카드

콘티는 사진이 설명하는 순서대로 세 섹션을 보여준다.

```text
핵심 장점 → 핏·코디 → 제품 확인
```

쇼핑몰 상단의 대표사진 영역(상품 갤러리)은 플랫폼 UI이므로 이 구조에 포함하지 않는다. 콘티의 `첫 장면`은 갤러리가 아니라 `핵심 장점` 안에 들어가는 생성 콘텐츠다. 상세페이지 전체 흐름을 말할 때는 위 세 사진 영역 뒤에 에디터의 `구매 정보`가 붙어 네 단계가 된다. 상의·하의·아우터·원피스는 이 상위 흐름을 공유하고, 의류 종류별 사진 조건과 생성 옵션만 달라진다(ADR-0005).

- `핵심 장점`은 첫 장면과 상품의 핵심 장점을 보여준다.
- `핏·코디`는 코디 활용, 핏, 실제 착용 느낌을 보여준다.
- `제품 확인`은 제품 전체 모습과 확인 가능한 디테일을 보여준다.
- `구매 정보`는 사진을 생성하는 일이 아니므로 콘티에 빈 사진 자리를 만들지 않는다. 사이즈표·소재·세탁·배송·교환·반품·필수 고지는 에디터에서 넣는다 (ADR-0005).

각 카드는 컷 종류 대신 **사진 목적**과 썸네일을 보여준다.

요건:

- 사진 목적은 `첫 장면 / 핵심 장점 / 코디 활용 / 핏 확인 / 실제 착용 느낌 / 제품 전체 / 디테일 / 직접 구성`이다.
- 호버 또는 선택 시 방향, 샷 종류, 대상 색상을 노출한다.
- 대상 색상은 입력 단계에서 선택한 색상 원으로만 표시한다.
- 좌측 그립으로 드래그해 순서를 변경한다.
- 위/아래 이동, 복제, 삭제 퀵액션을 제공한다.
- 블록 사이에 `블록 추가` 삽입 버튼을 제공한다.
- 내 이미지 블록은 직접 업로드한 이미지를 그대로 사용한다.
- 내 이미지는 블록 사이로 드래그해 삽입할 수 있다.
- 같은 장소로 묶인 카드는 핏·코디 섹션 안에서 배지로 표시한다. 같은 장소는 별도 섹션이 아니다.
- 카드가 다른 섹션으로 이동하면 그 섹션에 맞는 사진 목적을 다시 정한다. `직접 구성`은 어느 섹션에서든 쓸 수 있다.

### 8.4 사진 목적과 세부 옵션

사용자는 인스펙터에서 사진 목적을 고른다. 촬영 방식은 사용자 탭으로 보여주지 않고, 선택한 사진 목적에 맞춰 시스템이 내부에서 정한다. 생성예시는 그 방식으로 만들 수 있는 구도와 분위기를 보여준다 (ADR-0003~0005).

섹션별 사진 목적:

| 섹션 | 고를 수 있는 사진 목적 |
|---|---|
| 핵심 장점 | 첫 장면, 핵심 장점 |
| 핏·코디 | 코디 활용, 핏 확인, 실제 착용 느낌 |
| 제품 확인 | 제품 전체, 디테일 |

`직접 구성`은 새 AI 사진을 만들 때 고르는 별도 프리셋이 아니다. 현재 화면에서는 직접 올린 `내 이미지`와 이전 저장분처럼 정해진 사진 목적이 없는 카드에만 사용한다. 카드의 방향·샷·색상·생성예시를 바꿔도 기존 사진 목적은 유지되며, 사용자가 섹션 구성을 바꿨다는 표시는 별도 `sectionCustom` 값으로 구분한다.

착용 이미지의 옵션:

```text
방향: 정면 / 뒷면 / 사이드
샷 종류: 풀샷 / 중간샷
```

`실제 착용 느낌`은 거울 셀피 생성예시를 사용한다. 방향은 없고 샷은 풀샷·중간샷만 쓰며, 얼굴은 기본적으로 폰으로 가린다. 시스템은 내부에서 항상 `mirror` 레시피를 사용한다 (ADR-0004, ADR-0007).

제품 확인 이미지의 옵션:

```text
제품 전체: 고스트샷
디테일: 입력 사진에서 확인되는 핵심 부위
```

고스트샷은 모델 없이 옷 전체를 보여준다. 기본은 사람이 입은 듯한 부피와 실루엣이 있는 표현이며, 사용자가 고른 생성예시가 옷을 펼쳐 위에서 본 플랫레이 표현이면 같은 고스트샷 안에서 그 표현과 프레이밍을 따른다. 플랫레이는 별도 선택값이나 저장값이 아니다(ADR-0008).

행거샷은 제공하지 않는다. 디테일은 어느 색상이든 사용자가 올린 `Detail` 슬롯 사진이 하나 이상 있으면 선택할 수 있다. 목표 색상에 Detail이 없으면 기준색, 그다음 Detail 보유 첫 색상의 사진을 구조·재질 근거로 쓰고 색만 목표 색상군으로 전환한다. 앞면·뒷면·착용 이미지처럼 멀리 찍힌 사진에서 원단·안감·부자재를 추측해 만들지 않는다. **연결된 실제 생성예시의 목표**는 확인되는 핵심 부위만 상품당 1~3개 보여주고 근거가 없는 부위는 제외하는 것이다. 화면은 QC 승인 운영 카탈로그에서 조건별 최대 6장을 보여주며, 예시가 없는 조건은 임시 사진으로 채우지 않고 `준비 중`으로 표시한다(`TODO.md`).

상품이 아우터일 때는 AI가 만드는 착용 이미지에만 `아우터 열림 정도`를 추가한다. 섹션과 관계없이 사람이 아우터를 입고 있는 사진에 적용하며, 제품만 나온 사진과 내 이미지에는 적용하지 않는다.

```text
아우터 열림 정도: 전체 열림 / 부분 열림 / 전체 닫힘
```

기본값은 `전체 열림`이다. `부분 열림`의 정확한 단추·지퍼 위치는 상품 사진에서 확인되는 실제 구조 안에서 AI가 자연스럽게 정한다. 선택값은 생성예시의 열림 모습보다 우선하지만, 상품 사진에 없는 단추·지퍼·스냅·벨트·안감을 새로 만들거나 실제 잠금 구조를 바꾸면 안 된다. 아우터 여부는 생성예시 원본이 아니라 현재 프로젝트 상품의 의류 종류로 판단한다. 제품 확인 이미지·내 이미지와 아우터 이외 카테고리에는 이 항목을 표시하거나 적용하지 않는다.

### 8.5 분위기 예시와 레퍼런스

`분위기 예시` 영역은 샷 종류 아래에 위치한다.

요건:

- `생성예시`와 `내 레퍼런스` 세그먼트를 제공한다.
- 생성예시는 현재 섹션, 사진 목적, 방향, 샷 종류와 목표 상품의 의류 종류에 따라 갱신된다. 원본 상품 종류와 달라도 `applicableClothingTypes`에 목표 종류가 포함된 예시는 사용할 수 있다. `상의·아우터 공용`은 별도 사용자 설정이 아니라 예시 자산의 적용 범위다(ADR-0006).
- `전부`는 장소·조명·분위기·포즈·프레이밍·구성, `포즈만`은 자세만, `배경만`은 장소·조명·분위기만 참고한다. 포즈만 자산은 옷·배경이 없는 중립 전신 마네킹이며 원본 크롭을 복사하지 않는다. 풀샷·중간샷 프레이밍은 현재 카드의 `shot`이 정한다. `pose`·`bg` 전용 자산이 없으면 `all`로 대신하지 않고 해당 범위 선택을 제공하지 않는다(ADR-0009).
- 어느 범위에서도 예시 속 의류·신발·액세서리를 가져오지 않는다. 판매 상품과 사용자가 고른 매칭 의류가 모든 착용컷 의류의 유일한 근거다.
- 내 레퍼런스에서는 사용자가 참고 이미지를 업로드하고 삭제할 수 있다.
- 배경 선택은 콘티보드에서 제거한다. 공간·분위기는 생성예시와 같은 장소 묶음으로 조정한다.

### 8.6 생성예시와 매칭 의류 편집

착용 이미지의 자세와 구도는 생성예시로 정하고, 함께 입힐 옷은 모든 착용 카드의 `매칭 의류 편집`에서 조정한다.

요건:

- 포즈를 별도 목록으로 다시 고르게 하지 않는다. 선택한 생성예시의 자세를 참고한다.
- 매칭 의류는 분석 단계에서 선택된 항목을 기반으로 노출한다.
- 변경 사항은 다음 생성 단계에서 적용된다.

### 8.7 추가 옵션

세부 옵션 하단에 접이식 추가 옵션을 제공한다.

```text
모델 얼굴: 동일 / 노출 / 비노출
앵글: 동일 / 로우 / 하이
```

### 8.8 수정 잠금

사용자가 AI 컷을 수정 중인 상태에서 다른 카드를 선택하려 하면, 먼저 수정 완료를 요구한다.

요건:

- 우측 인스펙터에 `수정 완료를 먼저 눌러주세요` 경고를 표시한다.
- 하단에 `원래대로`와 `수정 완료` 2분할 액션을 제공한다.
- `원래대로`는 수정 전 스냅샷으로 복원한다.
- 내 이미지 블록은 AI 생성 옵션이 없으므로 같은 잠금 정책을 적용하지 않는다.

### 8.9 하단 액션바

콘티보드 하단에는 고정 액션바를 둔다.

구성:

- 이전
- AI 생성 컷 수와 셀러 사진 컷 수 요약
- 카피라이팅 ON/OFF 토글
- 생성 CTA

요건:

- 카피라이팅이 ON이면 AI가 상세페이지 카피를 자동으로 넣는다.
- 생성 CTA에는 예상 크레딧을 표시한다.
- 예상 크레딧은 `AI 생성 컷 수 x storyboardPerCut`으로 계산한다. 내 이미지(셀러 사진) 컷은 생성 작업이 없어 제외한다.

예시:

```text
상세페이지 생성하기 → 7 크레딧
```

## 10. 상세페이지 에디터

### 10.1 목적

에디터는 AI가 생성한 상세페이지를 최종 수정하는 전체 화면 작업 공간이다. 사용자는 블록 단위 상세페이지를 편집하고, 이미지와 텍스트를 추가 및 수정하며, 새 AI 이미지를 만들고 다운로드할 수 있다.

### 10.2 전체 구조

```text
상단 툴바
좌측 패널
중앙 캔버스
우측 미니프리뷰
모달 / 토스트 / 플로팅 레이어 패널
```

상단 툴바:

- wearless 워드마크
- AI, 의류, 이미지, 프레임, 텍스트, 오브젝트 탭
- 편집 중인 제품명
- Undo, Redo
- 미리보기
- 저장
- 다운로드

에디터에서는 일반 상단 네비게이션을 숨긴다.

### 10.3 캔버스와 블록

캔버스는 블록을 세로로 쌓는 상세페이지 편집면이다.

요건:

- 블록 기준 폭은 1000px이다.
- 초기 줌은 40%다.
- Ctrl 또는 Cmd + 휠로 10% 단위 확대/축소한다.
- 회색 빈 영역에 마우스를 올리면 줌 컨트롤을 표시한다.
- 각 블록은 고정 높이를 가진다.
- 요소가 블록 밖으로 넘치면 클립한다.
- 선택한 블록은 위/아래 pill 핸들로 높이를 조절한다.
- 상단 pill로 높이를 늘리거나 줄일 때 내부 요소 y 좌표도 함께 보정한다.

블록 퀵액션:

- 위로 이동
- 아래로 이동
- 빈 블록 추가
- 레이어 열기
- 이 블록 다운로드
- 블록 삭제

### 10.4 요소 선택과 조작

선택 가능한 요소:

- 이미지
- 텍스트
- 도형
- 선

요건:

- 선택 요소는 Sky bounding box와 핸들로 표시한다.
- 요소는 드래그 이동할 수 있다.
- 단일 선택 요소는 모서리/가장자리 리사이즈가 가능하다.
- 단일 선택 요소는 회전 핸들로 회전할 수 있다.
- 텍스트는 더블클릭으로 인라인 편집한다.
- 이미지가 비어 있는 프레임 슬롯은 `이미지 추가` 버튼을 표시하고, 클릭 시 의류 탭으로 이동한다.

스냅:

- x축 이동은 좌측, 중앙, 우측 기준에 자석 스냅한다.
- 회전은 0도, 90도, 180도, 270도 근처에서 스냅한다.

다중 선택:

- Shift + 클릭으로 다중 선택한다.
- 다중 선택 상태에서 드래그하면 그룹 이동한다.
- Delete 또는 Backspace로 선택 요소를 삭제한다.

### 10.5 Undo / Redo / 저장

요건:

- Ctrl 또는 Cmd + Z로 Undo를 실행한다.
- Shift + Ctrl 또는 Cmd + Z, 또는 Ctrl 또는 Cmd + Y로 Redo를 실행한다.
- Ctrl 또는 Cmd + S로 저장 토스트를 표시한다.
- 히스토리는 빠른 연속 변경을 하나의 단계로 합친다.
- 저장은 저장 버튼·자동 저장(디바운스)·이탈 시 플러시로 편집 상태를 영속화한다. 실제 서비스에서는 서버 저장과 동시 편집 충돌(리비전) 처리 정책이 필요하다.

### 10.6 레이어

레이어는 현재 블록의 요소 스택을 다룬다.

요건:

- 블록 퀵액션의 레이어 버튼을 누르면 우측에 플로팅 레이어 패널을 연다.
- 플로팅 패널은 헤더 드래그로 위치 이동할 수 있다.
- 레이어 목록은 위쪽이 최상단이다.
- 드래그로 레이어 순서를 바꿀 수 있다.
- 요소 표시/숨김 토글을 제공한다.
- 요소 잠금/잠금 해제 토글을 제공한다.
- 단축키 `[`는 뒤로, `]`는 앞으로 보낸다.
- 이미지/오브젝트/텍스트 패널에서도 앞으로, 뒤로, 맨 앞으로, 맨 뒤로 이동 버튼을 제공한다.

### 10.7 우측 미니프리뷰

우측 미니프리뷰는 상세페이지 전체 블록을 실제 비율 썸네일로 보여준다.

요건:

- 각 미니 블록은 실제 블록 비율을 유지한다.
- 클릭하면 해당 블록 위치로 스크롤한다.
- 드래그앤드롭으로 블록 순서를 변경한다.
- 선택 중인 블록을 강조한다.

### 10.8 AI 탭

AI 탭은 `새 이미지 추가`와 `현재 이미지 수정`을 제공한다.

새 이미지 추가:

- `첫 장면 / 핵심 장점 / 코디 활용 / 핏 확인 / 실제 착용 느낌 / 제품 전체` 중 사진 목적을 먼저 고른다. 상품 전체 색상 중 실제 `Detail` 입력이 하나 이상 있을 때 `디테일`도 보여준다. 새 이미지는 먼저 의류 탭에 들어가므로 현재 에디터 블록의 섹션에 선택지를 제한하지 않는다.
- 사진 목적을 바탕으로 시스템이 내부 생성 레시피를 정하고, 생성예시는 그 안에서 가능한 구도와 분위기를 고른다.
- 방향과 샷 종류는 콘티보드와 같은 규칙으로 필요한 항목만 보여준다.
- 분위기 예시와 내 레퍼런스를 제공한다.
- 색상은 입력 단계에서 선택한 색상 원만 노출한다.
- 모델은 접이식 영역에서 선택한다.
- 디테일 이미지는 상품 전체 색상 중 `Detail` 입력 사진이 하나 이상 있으면 만들 수 있다. 목표 색상에 없으면 타색 Detail의 구조·재질을 유지하고 색만 목표 색상군으로 전환한다.
- 생성 버튼에는 예상 크레딧을 표시한다.

현재 이미지 수정:

- 이미지 요소가 선택되어 있을 때만 활성 맥락을 제공한다.
- `비슷한 컷 만들기` 버튼을 제공한다.
- 변경 옵션으로 방향 변경, 샷 변경, 포즈 변경, 표정 변경, 배경 변경을 제공한다.
- 디테일로 바꾸는 옵션은 제공하지 않는다. 디테일은 상품 전체 중 실제 `Detail` 입력을 연결할 수 있는 `새 이미지 추가`에서만 만들며, 목표 색상에 없으면 타색 근거의 색만 전환한다. 서버도 수정 경로 우회 요청을 `detail_variation_unsupported`로 차단한다.
- 기존 이미지는 유지하고 새 이미지는 의류 탭에 추가한다.

### 10.9 의류 탭

의류 탭은 에디터에서 사용할 이미지 라이브러리다.

요건:

- 입력 단계의 색상 정보를 바탕으로 색상 그룹을 보여준다.
- 그룹은 접기/펼치기가 가능하다.
- 직접 이미지 업로드 버튼을 제공한다.
- 이미지를 클릭하면 캔버스에 삽입한다.
- 프레임 슬롯이 이미지 선택을 기다리는 경우, 클릭한 이미지를 해당 빈 슬롯에 채운다.
- 이미지 호버 시 `AI 편집` pill을 표시한다.
- 체크박스로 여러 이미지를 선택하고 삭제할 수 있다.
- 삭제는 의류 목록에서만 제거하며, 이미 캔버스에 들어간 이미지는 유지한다.

### 10.10 이미지 / 도형 / 선 속성 패널

이미지, 도형, 선은 같은 속성 편집 패널을 공유하되 제목과 일부 필드를 상황에 맞게 바꾼다.

공통:

- 가로/세로 크기 입력
- 비율 잠금
- 회전
- 둥근 모서리
- 레이어 이동

이미지:

- 채움 썸네일
- 투명도
- 크롭 진입 버튼

도형:

- 채움 색상
- 투명도
- 테두리 추가/제거
- 테두리 색상
- 테두리 굵기

선:

- 선 색상
- 굵기
- 점선 / 파선 / 실선 스타일

### 10.11 텍스트 탭

텍스트 탭은 텍스트 추가와 선택 텍스트 편집을 제공한다.

요건:

- 텍스트 추가 버튼과 `T` 단축키를 제공한다.
- 폰트 선택: Pretendard / Cal Sans / Roboto Mono
- 굵기 선택: Light / Regular / Medium / SemiBold / Bold
- 크기, 행간, 자간을 조절한다.
- 좌/중앙/우 정렬을 제공한다.
- Bold, Italic, Underline, Strike를 토글한다.
- 말머리 없음, 글머리 기호, 번호 목록을 제공한다.
- 글자 색상과 투명도를 조정한다.
- 하이라이트 색상 또는 없음 옵션을 제공한다.

### 10.12 프레임 탭

프레임은 새 블록으로 추가되는 템플릿이다.

프레임 목록:

```text
2분할
3컷 구성
FAQ
Before / After
컬러 비교
상품 정보 카드
```

요건:

- 프레임을 클릭하면 상세페이지 끝 또는 현재 맥락에 새 블록으로 추가한다.
- 프레임을 드래그하면 블록 사이에 삽입할 수 있다.
- 프레임 내부 이미지 칸은 빈 슬롯으로 생성되며, 의류 탭에서 이미지를 채운다.

### 10.13 오브젝트 탭

오브젝트 탭은 도형, 선, 블록 배경을 다룬다.

요건:

- 블록이 선택되어 있으면 블록 배경색 옵션을 보여준다.
- 블록 배경 옵션은 흰색, 연회색, 잉크를 제공한다.
- 기본 도형은 원, 사각형, 삼각형을 제공한다.
- 선은 왼쪽 화살표, 선, 오른쪽 화살표를 제공한다.
- 클릭하면 현재 블록 중앙에 추가한다.
- 드래그하면 원하는 블록 위치에 추가한다.

### 10.14 구매 정보와 사진 없는 내용

사진을 새로 만들 필요가 없는 표·글·안내는 콘티보드가 아니라 에디터에서 넣는다. 에디터의 `내용 추가` 목록은 아래 순서로 한 번에 보여준다. 사용자가 먼저 세 가지 유형 중 하나를 고르게 하지는 않는다.

| 중요도 | 내용 | 처음 열었을 때 |
|---|---|---|
| 반드시 확인 | 사이즈·실측, 소재 혼용률, 색상·옵션, 세탁법, 배송·교환·반품, 필수 고지 | 기본 페이지에 자리를 마련한다. 값이 없으면 `정보 입력 필요`로 표시한다. |
| 판매에 도움 | 핵심 장점 문구, 핏 설명, 모델 정보, 비침·신축성·두께, 컬러 설명 | 확인된 정보로 초안을 만들고 사용자가 수정하거나 삭제한다. |
| 필요할 때 추가 | 브랜드 이야기, FAQ, 리뷰, 관련 상품, 쿠폰·프로모션, SNS 안내 | `내용 추가` 목록 아래쪽에서 사용자가 고른다. |

사이즈·세탁·AI 생성 안내는 기존 자동 블록으로 만들고, 나머지는 에디터의 일반 정보 블록으로 만든다. 일반 정보 블록은 `infoType`으로 내용 종류만 구분하며 AI 사진 생성 대상이 아니다.

리뷰 블록은 사용자가 가진 실제 리뷰를 입력하는 자리다. AI가 구매자 후기나 평점을 지어내지 않는다.

사이즈 안내는 사용자가 입력한 실측 정보와 연동한다. 세탁 안내는 AI가 사실을 지어내지 않고 규칙 기반 초안으로 만든다. 소재나 케어라벨 근거가 부족하면 실제 상품 케어라벨 확인을 권장하는 문구를 포함한다. AI 생성 안내는 일부 이미지가 AI를 활용해 생성되었고 실제 색상과 핏이 촬영 환경 및 화면 설정에 따라 다를 수 있음을 알린다.

### 10.15 미리보기

미리보기는 전체 상세페이지를 블록 높이 그대로 이어 붙여 오버레이로 보여준다.

요건:

- 편집 핸들과 패널 없이 최종 상세페이지 형태만 보여준다.
- 닫기 버튼을 제공한다.

### 10.16 다운로드

다운로드는 모달로 제공한다.

옵션:

```text
전체 상세페이지 긴 PNG 1장
블록별 PNG ZIP
```

요건:

- 모달에서 형식을 선택한다.
- 현재 MVP에는 해상도 옵션을 두지 않는다.
- 다운로드 버튼을 누르면 다운로드 시작 토스트를 표시한다.
- 실제 파일 생성 및 ZIP 패키징은 백엔드 또는 클라이언트 렌더링 구현이 필요하다.

### 10.17 뒤로가기 제한

생성 완료 후에는 콘티보드로 자유롭게 되돌아가 생성 전 상태를 바꾸는 흐름을 허용하지 않는다. 필요한 컷 추가와 수정은 에디터에서 수행한다.

요건:

- 사용자가 콘티보드 단계로 돌아가려 하면 제한 모달을 표시한다.
- 모달 문구는 "이미 생성이 완료된 상세페이지입니다. 필요한 컷은 이 페이지에서 추가하거나 수정해주세요." 취지로 작성한다.

## 11. 자동 카피라이팅 정책

### 11.1 ON/OFF 위치

카피라이팅 ON/OFF는 콘티보드 하단 액션바에서 제공한다. 기본값은 ON이다.

### 11.2 적용 원칙

카피라이팅이 ON이면 AI는 상세페이지 블록에 필요한 문구를 자동으로 배치한다.

요건:

- 상품명, 소재, 강조 특징, 핏, 대상 성별, 매칭 의류 정보를 참고한다.
- 사용자가 입력하거나 확인한 정보를 우선한다.
- 확인되지 않은 소재, 세탁법, 기능성은 단정하지 않는다.
- 과장된 효능 표현을 피한다.
- 생성 후 사용자는 에디터에서 모든 텍스트를 직접 수정할 수 있다.

### 11.3 사진 목적별 카피 방향

첫 장면:

- 첫 화면에서 상품의 감정적 매력 또는 핵심 상황을 짧게 제시한다.

핵심 장점:

- 강조 특징을 바탕으로 1~3개 핵심 장점을 설명한다.

코디 활용:

- 착용 상황, 코디 활용, 분위기를 중심으로 작성한다.

핏 확인:

- 핏, 실루엣, 앞/뒤/사이드 구조 확인에 초점을 둔다.

실제 착용 느낌:

- 일상에서 어떻게 보이는지 설명한다. 실제 구매자의 리뷰나 착용 후기처럼 꾸며 말하지 않는다.

제품 전체:

- 전체 형태, 색상, 기본 구성을 사실 중심으로 설명한다.

디테일:

- 입력한 디테일 사진에서 실제로 확인되는 소재감, 마감, 부자재만 설명한다.

직접 구성:

- 사용자가 정한 제목과 목적을 우선하고, 확인되지 않은 사실은 덧붙이지 않는다.

사이즈 안내:

- 실측값과 측정 오차 안내를 포함한다.

세탁 안내:

- AI 카피가 아니라 규칙 기반 프리셋이다(의류 종류별 대표 소재 기준, 애매하면 기본 세탁방침 — §10.14).
- 실제 케어라벨 확인을 권장한다.

AI 안내:

- AI 생성 이미지 사용 사실과 실제 상품과 차이가 있을 수 있음을 알린다.

## 12. 크레딧 정책

### 12.1 기본 원칙

사용자가 비용을 인지하지 못한 상태에서 크레딧이 차감되지 않도록, 크레딧이 필요한 행동은 실행 전에 예상 소모량을 명확히 표시한다.

크레딧 예고가 필요한 행동:

- 마네킹 단일컷 최초 생성 또는 재생성
- 콘티보드에서 상세페이지 생성
- 에디터에서 새 이미지 생성
- 에디터에서 현재 이미지 수정

### 12.2 단가 설정

단가는 `creditCosts` 한 곳에서 관리한다.

```text
mannequinGenerate   // 마네킹 단일컷 생성·재생성 (=2)
mannequinAdjust     // @deprecated — fitProfile 재생성으로 통합, 단가 0
storyboardPerCut
editorImage
```

현재 mock 구현의 값은 정책 확정 전 임시값이다. 실제 서비스에서는 플랜, 무료 체험, 실패 환불, 재시도, 생성 품질 문제에 따른 크레딧 복구 정책을 별도로 정의해야 한다.

### 12.3 잔액 반영

크레딧을 소모하는 행동(마네킹 생성·재생성, 상세페이지 생성, 에디터 이미지 생성·수정)은 차감 후 잔액을 단일 소스로 갱신한다. 차감은 서버 책임이며, 소모 API는 `{ data, credits }` 봉투로 차감 후 잔액을 돌려주고 화면은 이를 받아 잔액 표시(네비게이션 배지·에디터)를 갱신한다. 프론트가 임의로 선차감하거나 자체 계산하지 않는다. (데이터 계약은 `common_data_contract.md` §6)

## 13. 데이터 및 상태 계약

### 13.1 프론트 상태 머신

URL 기준 라우트:

```text
/ → /create/input
/library
/create/input
/create/mannequin
/create/storyboard
/create/generating
/editor/:id
```

생성 플로우:

```text
input + inline analysis → mannequin → storyboard → generating → editor
```

`ProductInput` 안에서 분석을 인라인으로 처리하므로, 사용자가 경험하는 첫 단계는 `input + analysis`가 병합된 형태다. `generating`은 콘티 확정 후 에디터 진입 전 로딩 상태이며, `/editor/:id`는 상단 네비게이션 없이 독립 작업 공간으로 열린다.

전역 상태는 계정, 카탈로그, 생성 플로우의 주요 선택값만 유지한다. 폼 draft, hover, 로딩 phase, 패널 펼침 여부처럼 화면 안에서만 필요한 상태는 각 컴포넌트의 로컬 상태로 둔다.

### 13.2 엔티티·카탈로그·API 계약 (정본 위임)

엔티티(필드·타입·enum), 공통 카탈로그 목록, `lib/api` 함수 시그니처와 응답 shape의 **정본은 `common_data_contract.md`**다. PRD는 어느 시점에 봐도 계약과 어긋나지 않도록 이 목록을 중복 나열하지 않고 위임한다.

- 엔티티·enum 사전: `common_data_contract.md` §2~§4 (Project·ProjectSummary·Product·ColorGroup·ImageAsset·Measurement·Analysis·MannequinCut·StoryboardBlock·EditorBlock/Element·Wardrobe·GenJob·Account)
- 공통 카탈로그(`getCatalogs`) 목록: §5
- API 경계 함수·크레딧 봉투·유료 job 멱등 규약: §6
- 코드가 아직 계약을 못 따라간 부분(마이그레이션 갭): `TODO.md` §1

아래는 PRD 정체성으로 유지하는 데이터 경계 원칙이다(구체 필드·함수명은 위 계약이 정본):

- 화면 컴포넌트는 `src/lib/api/index.js`만 데이터 접근 경계로 사용한다. 실제 API·Supabase·스토리지·AI 엔드포인트를 직접 호출하지 않는다.
- 모든 API 함수는 비동기 Promise를 반환하고, 장시간 생성 작업은 진행률/단계 콜백을 제공한다.
- 실제 백엔드 전환 시 `src/lib/api/index.js`의 구현만 교체하고, 화면이 기대하는 함수 시그니처와 응답 shape는 유지한다.
- 실제 서비스의 각 API는 인증·프로젝트 id·파일 업로드·생성 job·실패/재시도·크레딧 트랜잭션을 포함한다(FastAPI·Supabase·R2 연동 — `backend_integration_plan.md`). 이 PRD는 서비스 요구사항과 계약 수준까지만 정의한다.

### 13.5 프론트엔드 구현 전제

현재 MVP 검증용 프론트엔드는 Vite + React SPA로 유지한다.

요건:

- React Router 라우트 구조는 생성 플로우, 보관함, 에디터의 화면 경계를 표현한다.
- JavaScript/JSX를 기본 언어로 유지하고, TypeScript는 공유 계약, store, API 계약처럼 안정성이 필요한 영역부터 점진 도입한다.
- 스타일은 CSS와 CSS 변수 토큰을 사용한다. 화면별 CSS 분리는 허용하되, 신규 UI에서는 인라인 스타일, CSS-in-JS, 토큰 없는 임의 색상을 사용하지 않는다.
- 에디터의 이동, 리사이즈, 회전 조작은 검증된 조작 라이브러리를 우선 사용한다.
- 실제 백엔드 연동 전까지 데이터는 mock layer를 통해서만 흐른다.

## 14. 디자인 시스템

### 14.1 방향

Wearless UI는 Cal.com 계열의 모노크롬 생산성 도구 미감을 기준으로 한다. 흰 캔버스, 잉크 텍스트, 절제된 그림자, pill CTA, 약한 오로라 배경을 사용한다. 색은 넓은 면적에 쓰지 않고 선택, 링크, 활성 상태, 생성 상태를 알리는 데 제한적으로 사용한다.

### 14.2 타이포그래피

폰트:

- Display: Cal Sans
- Body/UI: Pretendard Variable
- Mono: Roboto Mono

요건:

- 큰 제목과 카드 타이틀은 Cal Sans를 사용한다.
- 본문, 라벨, 버튼은 Pretendard를 사용한다.
- 코드성 숫자와 일부 영문 라벨은 Roboto Mono를 사용할 수 있다.
- CTA와 네비게이션은 과한 볼드 대신 500~540 weight 중심으로 둔다.

### 14.3 색상

기본 색상:

```text
Primary text: #0e0d14
Secondary text: #898989
Faint text: #b4b4b4
White surface: #ffffff
Soft surface: #f5f5f5
CTA bg: #2C2C2C
CTA hover: #1B1B1B
Link / accent: #0099ff
Active blue: #4f88c9
Selected sky ring: #8fbfee
```

배경 glow:

```text
Sky: #c3dcf6
Sage: #d8e8c6
Sun: #f8d978
Mauve: #ead0e3
```

요건:

- CTA는 검정에 가까운 warm near-black pill을 사용한다.
- Sky/Blue는 선택, 활성, 링크, 진행 상태에만 제한적으로 사용한다.
- 큰 UI 면과 카드 배경은 흰색 중심으로 유지한다.
- 경고와 삭제에는 별도 semantic red를 사용한다.

### 14.4 표면과 그림자

요건:

- 카드와 입력은 CSS border보다 ring shadow를 우선한다.
- 주요 카드에는 ring, diffuse, contact shadow를 조합한다.
- 배경 glow 위에 직접 텍스트를 놓지 않고 흰 surface 위에서 읽히게 한다.
- 일반 카드 radius는 12~16px 범위를 사용하되, 버튼과 토글은 pill radius를 사용한다.

### 14.5 공통 컴포넌트 패턴

- 버튼: primary / ghost / quiet / danger
- 아이콘 버튼: Lucide 라인 아이콘
- 칩: 선택 시 흰 배경 + Sky 테두리
- 세그먼트: 흰 pill 슬라이딩
- 언더라인 탭: 밑줄 슬라이딩
- 토글: iOS형 pill toggle
- 토스트: 우측 상단, 흰 배경, fade-out
- 모달: 중앙 overlay, Escape 닫기
- 스켈레톤: 로딩 상태 기본 표현
- 진행바: 완료 검정, 현재 blue 계열 강조

## 15. 검증, 에러, 안전 정책

### 15.1 로딩 상태

주요 비동기 작업은 로딩 상태를 반드시 제공한다.

- 제품 분석: 분석 스켈레톤
- 마네킹 생성: 3:4 의류 제작 애니메이션과 상태 문장(퍼센트·진행바 없음)
- 상세페이지 생성: 진행바와 체크리스트
- 보관함: 그리드 스켈레톤
- 에디터 초기 로딩: 중앙 로더

### 15.2 에러 상태

요건:

- 보관함 로딩 실패 시 에러 상태와 다시 시도 버튼을 제공한다.
- 분석, 생성, 다운로드 실패는 실제 서비스에서 오류 토스트, 재시도, 크레딧 복구 여부를 명확히 처리해야 한다.
- 상세페이지의 AI 이미지가 전부 실패하면 빈 페이지를 완료 처리하지 않는다. `all_cuts_failed`로 실패를 알리고 예약 크레딧을 돌려준다. 일부만 실패하면 성공 이미지로 에디터를 열고 실패한 자리만 다시 만들 수 있게 한다.
- 파일 업로드 실패는 실패한 파일명과 이유를 표시해야 한다.

### 15.3 확인이 필요한 행동

확인 모달 또는 경고가 필요한 행동:

- 마네킹컷 재생성
- 생성 완료 후 콘티보드 단계로 돌아가려는 행동
- 수정 중인 콘티 카드에서 다른 카드로 이동하려는 행동
- 크레딧 차감이 큰 생성 행동

### 15.4 신뢰와 고지

요건:

- AI가 생성한 이미지는 실제 상품과 차이가 있을 수 있음을 고지한다.
- 세탁 방법은 실제 케어라벨 확인을 우선하도록 안내한다.
- AI가 추정한 정보는 사용자가 수정할 수 있어야 한다.
- 실측은 AI 추정으로 채우지 않고 사용자가 직접 입력하게 한다.
